Sub-goal Detail

SG1 — Domain: Artifact + ChunkArtifactAssociation types done

Depends on: none

Objective: Define Artifact and ChunkArtifactAssociation domain types + invariants in apps/store/src/domain/ and domain/types/, mirroring chunk.ts/edge.ts.

  • Artifact carries id, uri, mimeType, createdAt, createdByStakeholderId; no domain operation mutates an existing artifact's uri/mimeType (IDEA-59) — only creation of a new Artifact is exposed.
  • ChunkArtifactAssociation carries chunkLabel, artifactId, branchId (nullable = mainline), originBranchId, status (active|superseded|deactivated), audit columns (IDEA-62). superseded is defined at the type level for IDEA-62 schema parity but this goal produces only active/deactivated rows — no write path in G08 ever sets superseded; a future goal introducing artifact-reference replacement (swap which artifact a chunk points to without a plain detach+attach) would own that behavior.
  • Unit test asserts no code path allows mutating an existing artifact's blob reference in place.

Verification: pnpm test:store

SG2 — Persistence: tables, repositories, blob store done

Depends on: SG1

Objective: Migration 0009 creates artifacts (id, uri, mime_type, created_at, created_by_stakeholder_id) and chunk_artifacts (id, chunk_label, artifact_id, branch_id nullable, origin_branch_id, status, created_at, created_by_stakeholder_id, updated_at, updated_by_stakeholder_id) plus IDEA-64's partial unique index. Add ArtifactRepository (create/findById) and ChunkArtifactRepository with findEffectiveForChunk(chunkLabel, branchId) — a set-based overlay read (a chunk can hold multiple artifacts per IDEA-60) returning { artifactId, branchId, status } per pair: within each scope (mainline rows where branch_id IS NULL; this branch's rows where branch_id = branchId), the most-recently-created row (by created_at, tie-broken by id) for a given artifact_id is authoritative — older rows for that same pair in that same scope are historical and ignored (every write inserts a new row rather than updating one, per IDEA-59/IDEA-62's append-only shape). The branch-scope's authoritative row for a pair wins over the mainline-scope's authoritative row for that same pair; the method returns one {artifactId, branchId, status} tuple per artifact_id whose winning row has status='active', with branchId reflecting which scope won (null = mainline). This is scoped only to chunk_artifacts — it does not attempt the generic chunks/edges overlay read still out of scope per chunk.repository.ts:154-159. Add ArtifactBlobStore interface + LocalFileBlobStore implementation backed by a new Docker named volume, per explicit user direction (local storage, swappable interface).

  • Migration creates both tables + partial unique index (chunk_label, artifact_id) WHERE branch_id IS NULL AND status='active'.
  • ArtifactRepository.create writes the blob via ArtifactBlobStore.put then inserts one metadata row; no update path exists.
  • ChunkArtifactRepository.findEffectiveForChunk(label, branchId) returns the correct set of {artifactId, branchId, status} tuples for three cases, each integration-tested against containerized Postgres: (a) mainline has A+B, branch has no rows for the chunk → returns {A:null,B:null}; (b) mainline has A+B, branch deactivates A → returns {B:null} (A's branch-scoped deactivated row wins and masks it); (c) mainline has A, branch adds C as active → returns {A:null, C:branchId}.

Verification: pnpm test:store

SG3 — API: branch-scoped artifact writes done

Depends on: SG2

Objective: ArtifactsController exposes POST /artifacts (upload: body carries base64 content + mimeType; creates one immutable Artifact) and POST /chunks/:label/artifacts (attach: body { artifactId, branchId?, stakeholderId }). DELETE /chunks/:label/artifacts/:artifactId?branchId=&stakeholderId= (detach) inserts a branch-scoped deactivated row rather than removing anything, mirroring IDEA-32's chunk-deactivation-row pattern. branchId is carried as a required query parameter (not a request body) since DELETE bodies are non-standard in this codebase's existing controllers.

  • POST /artifacts returns 201 with the created artifact's id + uri metadata (never raw bytes in the response).
  • POST /chunks/:label/artifacts returns 201 with the created association; 404 if the chunk label or artifact id does not exist.
  • DELETE .../artifacts/:artifactId?branchId=&stakeholderId= inserts a deactivated row; 400 if branchId query parameter is omitted (mainline cannot be detached from directly).

Verification: pnpm test:store

SG4 — API: effective reads + signed retrieval done

Depends on: SG3

Objective: GET /chunks/:label/artifacts?branchId= returns the effective (branch-overlaid, set-based per SG2) list of {artifactId, branchId, status} tuples for currently-active associations of a chunk via findEffectiveForChunk — branchId in each returned tuple reflects which scope's row won (null = mainline), so callers (including SG7's Docker verification) can directly observe whether an association is mainline-effective without inspecting the database. GET /artifacts/:id/download-token issues a time-limited HMAC-signed download token (reusing signHmacToken/verifyHmacToken from apps/store/src/auth/hmac-token.ts) instead of an S3 presigned URL. GET /artifacts/content/:token validates that token and streams the blob back from LocalFileBlobStore.

  • GET /chunks/:label/artifacts?branchId= returns the full set of currently-active {artifactId, branchId, status} tuples for the chunk, correctly overlaying branch-scoped rows over mainline per SG2's three test cases.
  • GET /artifacts/:id/download-token issues a token that expires (e.g. 5 minutes).
  • GET /artifacts/content/:token rejects an expired or tampered token (InvalidHmacTokenError) and, given a valid token, returns the exact bytes originally uploaded.

Verification: pnpm test:store

SG5 — Merge integration: chunk_artifacts in atomic merge done

Depends on: SG4

Objective: Extend BranchRepository.merge's atomic transaction (apps/store/src/persistence/branch.repository.ts:304-392) to also promote branch-scoped chunk_artifacts rows to mainline (branch_id -> NULL). Per SG2's authoritative-row rule, a branch can accumulate multiple historical rows for the same (chunk_label, artifact_id) pair (e.g. attach → detach → attach); promoting every such row would create duplicate mainline rows and violate IDEA-64's partial unique index. Merge therefore promotes only the authoritative (most-recently-created) branch-scoped row per (chunk_label, artifact_id) pair — all older same-pair branch rows are left as-is (branch-scoped, non-authoritative history; they do not migrate to mainline). The merge rejects (kind:'conflict') when an authoritative branch row is active and that exact (chunk_label, artifact_id) pair is already active on mainline, using the same identity-collision scope (not a full three-way divergence diff) as the existing chunk-label/edge-identity checks at lines 323-363. An authoritative branch row that is deactivated never conflicts (that is the mechanism by which a detach reaches mainline) and is promoted unconditionally. This closes the gap flagged in review: without promoting deactivated rows too, a branch-scoped detach would never take effect on mainline after merge. IDEA-46 names chunk-artifact-association modifications as in-scope for merge conflict detection; the specific identity-collision mechanics (vs. a full three-way divergence diff) are a repo-local scoping carried forward from G06's OQ2 resolution for chunks/edges, applied here by extension — not a verbatim restatement of IDEA-46's text.

  • Merging a verified branch with an authoritative added active chunk_artifacts row promotes only that row to branch_id=NULL on mainline.
  • Merging a branch with an authoritative deactivated chunk_artifacts row (a detach) promotes only that row to branch_id=NULL on mainline unconditionally, so the chunk's mainline effective-artifact set (SG4) no longer includes it afterward.
  • Merging a branch with older, non-authoritative same-pair rows (e.g. an earlier attach superseded by a later detach in the same branch) does not promote those older rows — only the authoritative one migrates, so no duplicate mainline row is ever created.
  • Merging a branch whose authoritative new active association would collide with an existing active mainline (chunk_label, artifact_id) pair returns kind:'conflict' and rolls back the entire transaction (zero rows mutated).

Verification: pnpm test:store

SG6 — MCP: upload-artifact, attach-artifact-to-chunk done

Depends on: SG5

Objective: Add upload-artifact and attach-artifact-to-chunk MCP tools in apps/mcp/src/tools, calling the store's POST /artifacts / POST /chunks/:label/artifacts as a delegated actor (IDEA-9), consistent with the capture-chunk/create-edge precedent. Given the MCP server's existing ~1MB in-memory request-body cap and JSON-only transport, upload-artifact accepts inline base64 content up to that cap only; larger binary artifacts are explicitly out of scope for this goal.

  • upload-artifact forwards base64 content + mimeType to POST /artifacts and returns the created artifact id.
  • upload-artifact surfaces a clear client-facing error (not a crash) when decoded content exceeds the store's configured max artifact size.
  • attach-artifact-to-chunk forwards { chunkLabel, artifactId, branchId, stakeholderId } to POST /chunks/:label/artifacts and returns the created association.

Verification: pnpm test:mcp

Open Questions

None. Large (over the MCP server's body-size cap) binary artifact ingestion is out of scope for this goal by design, not unresolved ambiguity: only inline/base64 content up to that existing cap is supported via MCP; direct HTTP upload (SG3) has no such cap beyond NestJS's body parser limits.

IDEA-61's storage-backend/signed-URL ambiguity (originally flagged as a gap) is resolved: IDEA-84 (gap report) and IDEA-85 (resolution — local file storage on a Docker volume behind a swappable ArtifactBlobStore interface, HMAC-signed download tokens) are both now promoted in Meridian. The IDEA-85 → IDEA-61 edge was inserted directly via psql into the Meridian Postgres container per explicit user direction, after meridian-create-edge repeatedly 404'd for that specific pair.
