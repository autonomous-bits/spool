Sub-goal detail

SG0 — persistence/infra: branches schema migration done

Depends on: none

Objective: Add the authoritative branches table and wire chunks to it for real.

  • Migration creates branches(id, name, discipline, status, submitted_at, diverged_at, created_at, updated_at, merged_at, origin_suggestion_id, created_by_stakeholder_id, merged_by_stakeholder_id) exactly per IDEA-31's authoritative schema (status IN ('draft','submitted','verified','merged'); diverged_at defaults to NOW(); created_by_stakeholder_id is NOT NULL REFERENCES stakeholders(id); merged_by_stakeholder_id nullable FK to stakeholders).
  • origin_suggestion_id is declared as a plain nullable UUID with no FK (the suggestions table does not exist in this repo yet), mirroring the exact scope-deviation precedent documented in migration 0001 for chunks.branch_id in G01. This deviation must be documented in the new migration's header comment.
  • CREATE UNIQUE INDEX idx_active_branch_name ON branches (name) WHERE status != 'merged' is created per the authoritative schema.
  • The existing chk_chunks_branchless_only_g01 CHECK constraint on chunks is dropped.
  • chunks.branch_id gets a real REFERENCES branches(id) ON DELETE CASCADE FK; chunks.origin_branch_id gets a real REFERENCES branches(id) FK (no cascade), both per the authoritative schema.
  • All existing G01 migrations/tests that assert branchless-only behavior continue to pass unmodified for the branchless path (IDEA-78 is not being revoked).

Verification: pnpm test:store (migration + repository e2e tests against containerized Postgres per the constitution).

SG1 — domain: Branch entity done

Depends on: none

Objective: Model Branch and its divergence point as a validated domain entity, reusing the existing Discipline vocab.

  • BranchStatus type models all four values (draft | submitted | verified | merged), matching the full CHECK constraint, even though this goal's write path only ever produces draft (mirrors ChunkStatus's precedent from G01).
  • Branch entity requires non-blank name, a valid Discipline, and a non-blank createdByStakeholderId; rejects blank/invalid input by throwing, mirroring Chunk's constructor-validation style.
  • DivergencePoint value type (opaque validated ISO-8601 timestamp) per IDEA-74/IDEA-41, defaulting to "now" at construction.
  • Unit tests cover: valid construction, blank name rejection, invalid discipline rejection, blank stakeholder rejection.

Verification: pnpm test:store (unit tests for the new domain module).

SG2 — API: Branches controller/service done

Depends on: SG0, SG1

Objective: Expose POST /branches and GET /branches/:id, per IDEA-24/IDEA-52/IDEA-34.

  • POST /branches accepts { name, discipline, stakeholderId } (creator attribution required, matching the CreateChunkRequest convention of requiring an existing stakeholderId rather than inventing one) and returns the created branch (id, name, discipline, status='draft', divergedAt, createdAt, updatedAt, createdByStakeholderId).
  • Returns HTTP 400 for missing/blank name/stakeholderId or invalid discipline, mirroring parseCreateChunkRequest's IO-boundary-guard style.
  • GET /branches/:id returns the same branch response shape, 404 if not found.
  • Persistence: a BranchRepository following the existing ChunkRepository pattern (parameterized queries only, per nestjs-security conventions).

Verification: pnpm test:store (controller/service unit tests + new e2e spec against containerized Postgres).

SG3 — branch-scoped chunk authoring done

Depends on: SG2

Objective: Let POST /chunks attach a chunk to an existing draft branch, without breaking the ratified branchless path (IDEA-78).

  • CreateChunkRequest gains an optional branchId. When present: the service loads the branch, returning 404 if missing; if the branch exists but branch.discipline !== request.discipline (single-discipline bound, IDEA-40/IDEA-53) or branch.status !== 'draft' (write-lock, IDEA-53), the service returns 409. Enforcement lives in the domain/service layer, not controller-only logic — consistent with the repo's existing practice pending Meridian's still-open IDEA-70 resolution on gateway-vs-domain placement.
  • When branchId is present, the persisted chunk row sets both branch_id and origin_branch_id to that branch's id (first-authored-here semantics).
  • When branchId is omitted, behavior is byte-for-byte unchanged from G01 (branch_id/origin_branch_id NULL, existing indexes apply).
  • ChunkResponse gains nullable branchId/originBranchId fields so branch attachment is observable over the API (addresses the read-side gap flagged in this goal's rubber-duck review).
  • GET /chunks/:id reflects the same fields; no separate endpoint needed.

Verification: pnpm test:store (unit + e2e covering both the branch-scoped and branchless paths).

SG4 — MCP exposure open

Depends on: SG3

Objective: Let an ideation assistant create a branch and author chunks within it, per IDEA-9.

  • New create-branch MCP tool mirrors capture-chunk's conventions exactly: validates untrusted input shape only (name/discipline/stakeholderId as non-empty strings), forwards to POST /branches, never invents a stakeholder identity, surfaces the store's 4xx errors as a typed validation error.
  • capture-chunk tool's input/parse function gains an optional branchId string field, forwarded as-is to the store; vocabulary/invariant validation remains the store's responsibility (not re-validated in the tool), per the existing tool's own documented convention.
  • Unit tests for both tools cover: valid input parsing, missing-field rejection, and successful/error forwarding (mocked fetch).

Verification: pnpm test:mcp.

SG5 — Docker end-to-end verification open

Depends on: SG4

Objective: Prove the full four-layer slice (MCP → API → domain → persistence) works against the real containerized system, not just in-process tests.

  • docker compose up --build spoolstore brings up the store against real Postgres.
  • Over real HTTP: POST /branches creates a draft branch; POST /chunks with that branchId succeeds and the response's branchId/originBranchId match; GET /branches/:id and GET /chunks/:id both reflect the association.
  • Also over real HTTP: a branchless POST /chunks (no branchId) still succeeds exactly as in G01, proving no regression.
  • A negative case is exercised: POST /chunks with a branchId from a different discipline, or a non-existent branchId, is rejected with a 4xx.
  • The MCP layer is exercised too, against the running store: the create-branch tool creates a branch, and the capture-chunk tool with that tool-created branch's id captures a chunk into it; both are confirmed via HTTP readback (GET /branches/:id, GET /chunks/:id).

Verification: manual/scripted HTTP exercise (curl or equivalent) plus MCP tool invocation against the running spoolstore container — in-process/unit tests alone do not satisfy this step.
