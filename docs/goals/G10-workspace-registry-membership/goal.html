G10 — Workspace Registry & Membership Bootstrap done

Size: 4/5 — two new domain concepts (Workspace, WorkspaceMembership), a new Postgres schema/migration, and a new NestJS API layer reusing (not extending) existing human-auth; no MCP layer, no isolation enforcement, so it stops short of 5.

Scope framing: this goal delivers the workspace registry + membership primitives (IDEA-94/95/96) only. It does not implement workspace isolation (IDEA-90), request-scoping/auth-claim binding (IDEA-92), or domain-invariant enforcement of membership boundaries (IDEA-98). After this goal, a workspaces table and membership rows exist and are administrable via API, but every existing chunk/branch/edge/artifact/suggestion/signal/notification path remains globally scoped exactly as before — Workspace has no isolation effect yet. Treat IDEA-88/89 as motivating product context, not as fully realized by this slice.

Sub-goal Detail

SG1 — Domain model done

Layer: Domain  |  Depends on: none

Objective: Add Workspace and WorkspaceMembership domain types plus invariants in apps/store/src/domain/ (e.g. workspace.ts, workspace-membership.ts), following existing domain-file conventions (see branch.ts, branch-lifecycle.ts). The domain layer owns pure decision logic only (what membership rows a creation implies, who may add a member); SG2's repository is responsible for executing those decisions as real writes — SG1 does not touch persistence.

Acceptance criteria:

  • Workspace has: id, name, createdByStakeholderId, createdAt, updatedAt — matching IDEA-96 exactly (see SG2's exact DDL).
  • WorkspaceMembership has: workspaceId, stakeholderId, createdAt — no role field, per IDEA-95.
  • A pure domain function (e.g. deriveInitialMembership(workspace)) returns the single membership row implied by a new workspace (the creator, per IDEA-88/95) — this is the authoritative statement of "creator becomes first member"; SG2's repository calls it and persists the result in the same transaction as the workspace insert, it does not re-derive the rule itself.
  • A pure domain function (e.g. assertCanAddMember(actorIsMember: boolean)) throws unless the caller is an existing member; the creator has no special privilege beyond being the first membership row (per IDEA-88/95).
  • Unit tests cover: deriveInitialMembership produces exactly one membership row for the creator, assertCanAddMember rejects a non-member caller, assertCanAddMember allows any existing member regardless of whether they were the creator.
  • Domain-level duplicate-membership handling: adding a member who is already a member is a distinct outcome from "non-member cannot add" — model it as its own explicit domain error (e.g. WorkspaceMembershipAlreadyExistsError), not silently swallowed and not conflated with the non-member-caller rejection. SG3 maps this domain error to HTTP 409 (see SG3).

Verification: pnpm test:store (targeted: new domain spec files)

SG2 — Persistence done

Layer: Persistence  |  Depends on: SG1

Objective: Add migration 0012_add_workspaces_tables.sql creating workspaces and workspace_memberships exactly per IDEA-96, plus a WorkspaceRepository (or equivalent) in apps/store/src/persistence/ providing: create-workspace-with-first-member (single atomic transaction, calling SG1's deriveInitialMembership), add-member (calling SG1's assertCanAddMember and raising SG1's WorkspaceMembershipAlreadyExistsError on duplicates), and an isMember(workspaceId, stakeholderId) lookup.

Exact schema (authoritative — do not deviate):

CREATE TABLE workspaces (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  created_by_stakeholder_id UUID NOT NULL REFERENCES stakeholders(id),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE workspace_memberships (
  workspace_id UUID NOT NULL REFERENCES workspaces(id),
  stakeholder_id UUID NOT NULL REFERENCES stakeholders(id),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (workspace_id, stakeholder_id)
);

Match existing migration conventions (see 0001_init_chunk_capture_schema.sql/0003_add_branches_table.sql) for exact column-type/default idioms (e.g. the repo's existing UUID-default and timestamp-default style) if they differ from the above — the above is the column/constraint contract from IDEA-96, not a mandate to break from repo SQL style.

Acceptance criteria:

  • Migration runs cleanly against the existing Postgres migration chain (0001-0011) with no FK/index conflicts.
  • Creating a workspace inserts exactly one workspaces row and one workspace_memberships row (creator) atomically — a failure in either rolls back both.
  • Adding a duplicate member raises SG1's WorkspaceMembershipAlreadyExistsError (backed by the composite primary key uniquely enforcing one row per (workspace, stakeholder)) — this is a distinct, explicit conflict outcome, not a silent no-op and not conflated with the non-member-caller rejection.
  • This goal does not add a workspace_id column, index, or constraint to any existing table (chunks, edges, branches, artifacts, chunk_artifacts, suggestions, verification_signals, feedback_notifications) — that is IDEA-90, explicitly out of scope (OQ1).
  • Integration tests run against a real containerized Postgres (per repo convention — no in-memory substitute).

Verification: pnpm test:store (targeted: new persistence integration spec)

SG3 — API done

Layer: API  |  Depends on: SG2

Objective: Add a WorkspacesModule (controller + service + DTOs, mirroring branches module structure) exposing POST /workspaces and POST /workspaces/:id/members per IDEA-94, wired into app.module.ts. Both endpoints require a valid existing human session token (reuse SessionTokenService exactly as branches/verification endpoints do — no new auth logic, no workspaceId claim, no X-Workspace-Id header). POST /workspaces: any authenticated human stakeholder may call it. POST /workspaces/:id/members: caller must be an existing member of the named workspace (checked via SG2's isMember), naming a target stakeholder id to add.

Acceptance criteria:

  • Request/response contract: POST /workspaces body { "name": string } → 201 { "id": uuid, "name": string, "createdByStakeholderId": uuid, "createdAt": ISO-8601 }. POST /workspaces/:id/members body { "stakeholderId": uuid } → 201 { "workspaceId": uuid, "stakeholderId": uuid, "createdAt": ISO-8601 }.
  • Missing/invalid session token on either endpoint → 401.
  • Valid token, POST /workspaces → 201 with created workspace id; caller is the sole initial member.
  • :id in POST /workspaces/:id/members naming a nonexistent workspace → 404.
  • Valid token, caller not a member of the target workspace, POST /workspaces/:id/members → 403.
  • Target stakeholderId in the request body naming a nonexistent stakeholder → 404.
  • Valid token, caller is a member, POST /workspaces/:id/members with a valid target stakeholder id → 201 (membership resource created) with membership confirmation.
  • Valid token, caller is a member, target stakeholder is already a member (SG1's WorkspaceMembershipAlreadyExistsError, SG2) → 409.
  • No MCP tool is added for either endpoint (both are human-only actions per IDEA-94, matching the established repo precedent that branch submit/verify/merge — also human-only — have no MCP tool exposure).
  • Controller + e2e-spec tests cover all cases above, including the non-member-rejection case (this is the one negative case that cannot be reproduced in the single-fixture Docker environment — see SG4).

Verification: pnpm test:store

SG4 — Docker end-to-end (mandatory final step) done

Layer: Docker e2e  |  Depends on: SG3

Objective: Bring up the real containerized system and exercise both new endpoints over HTTP using a real human session token obtained the only way the constitution permits.

Acceptance criteria:

  • Run docker compose up --build spoolstore (rebuild required — host dist/ must be rebuilt first per repo convention).
  • Obtain a real human session token via pnpm dev:session-token (drives the actual GET /auth/github/login → GET /auth/github/callback flow against the running container via the wired github-oauth-stub service). Do not substitute a hand-crafted or client-supplied token.
  • Using that token, call POST /workspaces → observe 201 and a created workspace id.
  • Using that same token, call POST /workspaces/:id/members naming the existing seeded bootstrap stakeholder (migration 0002_seed_bootstrap_stakeholder.sql, id 00000000-0000-0000-0000-000000000002) as the target → observe success and a persisted second membership row.
  • Negative case exercised in Docker: call POST /workspaces with no/invalid token → observe 401. (The other negative case — non-member rejected from add-member — is proven in SG3's Vitest e2e-spec, not here, since only one OAuth-mapped fixture identity exists in the Docker environment; do not attempt to fabricate a second real human token to reproduce it in Docker.)
  • Confirm persisted state directly against the running Postgres container (e.g. docker compose exec psql query on workspaces/workspace_memberships) as supplementary confirmation alongside the API responses.

Verification: manual Docker exercise as described above (not satisfied by unit/integration tests alone).

Open Questions

  • OQ1 (scope boundary, intentional): IDEA-90 (workspace_id columns + re-keyed uniqueness on chunks/edges/branches/artifacts/chunk_artifacts/suggestions/verification_signals/feedback_notifications) is explicitly deferred to a future goal. This goal's tables and existing tables remain unlinked.
  • OQ2 (scope boundary, intentional): IDEA-92 (session-token workspaceId claim, X-Workspace-Id header, OAuth callback workspaceId param) and IDEA-98 (Domain Invariant Protection Service membership-boundary amendment) are explicitly deferred to a future goal. This goal's endpoints authenticate with the existing, unmodified human session token only.
  • OQ3 (resolved for future goals, not implemented here): IDEA-99 (draft, unratified) originally flagged that IDEA-92's OAuth bootstrap path was ambiguous and that IDEA-92's "every request/MCP call needs X-Workspace-Id" conflicted with today's tokenless delegated routes. Both gaps are now resolved by promoted ADRs IDEA-101 (bootstrap: workspace-less session token, usable only for POST /workspaces, minted when a stakeholder has zero memberships) and IDEA-100 (mixed auth: delegated/tokenless routes validate X-Workspace-Id against the caller-supplied stakeholderId's membership row, no token required). This goal does not implement either resolution — it still authenticates POST /workspaces with today's unmodified IDEA-81 token, since IDEA-92 itself remains unimplemented — but a future goal implementing IDEA-92 can proceed directly against IDEA-100/101 without further Meridian escalation.
  • OQ4 (cross-goal note, informational only): G09's notification fan-out was justified as global specifically because "no membership model exists" at the time. That statement becomes literally stale once this goal lands `workspace_memberships`, even though notification behavior itself is intentionally untouched here. A future workspace-scoping goal should revisit G09's fan-out scope alongside IDEA-90/92/98.
