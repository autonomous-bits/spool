Sub-goal Detail

SG0 — Persistence/infra: DB bootstrap & migration open

Depends on: none

Objective: Add a repo-local migration extending the chunks table with chunk_type and context_kind columns and wire it into Postgres startup; seed one bootstrap stakeholder row for dev/test use.

  • A SQL migration file exists under apps/store (e.g. apps/store/migrations/) creating, at minimum: stakeholders(id UUID PK, name, email, role, discipline, created_at) per schema.sql, and chunks(id UUID PK, label, content, discipline, status, branch_id, origin_branch_id, created_by_stakeholder_id, updated_by_stakeholder_id, created_at, updated_at) per schema.sql plus the two columns ratified by IDEA-77's amendment to IDEA-31: chunk_type VARCHAR(50) NOT NULL CHECK (chunk_type IN ('feature','capability','constraint','adr','spike')) and context_kind VARCHAR(50) NOT NULL CHECK (context_kind IN ('permanent','transient')), plus the uniqueness index ratified by IDEA-78's amendment to IDEA-31/IDEA-32: CREATE UNIQUE INDEX idx_chunks_draft_mainline ON chunks (label) WHERE branch_id IS NULL AND status = 'draft';.
  • The migration is applied automatically both for the Dockerized Postgres (via compose.yaml init) and for host-side integration tests against localhost:5433.
  • Exactly one bootstrap stakeholder row is seeded (fixed, documented UUID) usable by later sub-goals and the Docker e2e step.

Verification: pnpm test:store (migration-runner unit/integration test).

SG1 — Domain: Chunk entity & vocab open

Depends on: none (parallelizable with SG0)

Objective: Implement a rich Chunk domain entity plus closed ChunkType, ContextKind, and Discipline value types with validation invariants, per IDEA-72/IDEA-73/IDEA-4/IDEA-14.

  • ChunkType accepts exactly: feature, capability, constraint, adr, spike — anything else throws a domain validation error.
  • ContextKind accepts exactly: permanent, transient.
  • Discipline accepts exactly: product, architecture, design, engineering, security, governance.
  • Chunk construction rejects empty/blank label or content.
  • Chunk carries createdByStakeholderId (required) for authoring traceability (IDEA-11).

Verification: pnpm test:store (unit tests for entity + all three vocab types, including rejection cases).

SG2 — Persistence: ChunkRepository open

Depends on: SG0, SG1

Objective: Implement a Postgres-backed ChunkRepository with create(chunk) and findById(id), mapping domain invariants to/from the extended chunks table.

  • create persists a chunk with status = 'draft', branch_id = NULL, and the new chunk_type/context_kind columns.
  • findById returns the persisted chunk with all fields round-tripped exactly (including vocab and discipline), or an explicit not-found result.
  • Integration tests run against the containerized Postgres per docs/specifications testing convention (no in-memory substitute).

Verification: pnpm test:store (Vitest integration test against containerized Postgres on localhost:5433).

SG3 — API surface: ChunksController open

Depends on: SG2

Objective: Expose POST /chunks and GET /chunks/:id on the NestJS store, wired through a ChunkService to SG2's repository, per IDEA-52/IDEA-34.

  • POST /chunks accepts { label, content, discipline, chunkType, contextKind, stakeholderId }, returns 201 with the created chunk including generated id.
  • POST /chunks returns a 400 with a validation error for any invalid vocab value, missing required field, or unknown stakeholderId.
  • GET /chunks/:id returns 200 with the chunk for a known id, and 404 for an unknown id.

Verification: pnpm test:store (Vitest e2e test under apps/store/test/*.e2e-spec.ts against containerized Postgres).

SG4 — MCP exposure: capture-chunk tool open

Depends on: SG3

Objective: Add an MCP tool (e.g. capture-chunk) in apps/mcp that calls the store's POST /chunks over HTTP on behalf of a human stakeholder, acting as a delegated actor per IDEA-9.

  • Tool accepts the same fields as POST /chunks and forwards them to HARNESS_URL.
  • Tool surfaces the store's validation errors (400) back to the calling agent without swallowing them.
  • Tool response includes the created chunk's id on success.

Verification: pnpm test:mcp (unit/integration test with a stubbed or real store HTTP call).

SG5 — Docker end-to-end verification open

Depends on: SG4

Objective: Bring the store up in Docker (per apps/store/AGENTS.md, the store always runs via Compose, never directly on host) and exercise the real capability end to end, not just in-process tests. The MCP server is a separate Node process (node apps/mcp/dist/main.js, not part of compose.yaml) started with HARNESS_URL pointed at the running containerized store.

  • docker compose up --build spoolstore starts Postgres (migrated per SG0) and the store container successfully, store reachable on localhost:3000.
  • A real HTTP POST /chunks against localhost:3000 followed by GET /chunks/:id returns the same chunk, including chunkType/contextKind/discipline round-tripped correctly.
  • Separately, apps/mcp is built and started as its own Node process with HARNESS_URL=http://localhost:3000; a real capture-chunk tool call against that running MCP process creates a chunk retrievable via the store's GET /chunks/:id on localhost:3000.

Verification: Manual/scripted exercise — docker compose up --build spoolstore for the store/Postgres, plus a separately started apps/mcp process, plus real HTTP/MCP calls against both. Not satisfied by unit or e2e-in-process tests alone.

Open Questions

  • Schema gap — RESOLVED via Meridian IDEA-77 (promoted ADR amendment to IDEA-31): IDEA-31's authoritative schema now formally includes chunk_type/context_kind columns on chunks, matching IDEA-72/IDEA-73's vocabulary exactly. SG0's migration implements this ratified schema directly (no longer a stopgap).
  • Branchless draft retrieval — RESOLVED via Meridian IDEA-78 (promoted ADR amendment to IDEA-31/IDEA-32): branchless/draft chunk capture (branch_id = NULL, status = 'draft') is now formally ratified as a first-class path, distinct from branch-based authoring. SG0's migration should add the accompanying uniqueness index: CREATE UNIQUE INDEX idx_chunks_draft_mainline ON chunks (label) WHERE branch_id IS NULL AND status = 'draft';. This goal retrieves chunks by generated id, not by label, so it did not depend on this ruling either way, but the ratification is now available for any future label-based retrieval work.
  • Stakeholder registration is out of scope for this goal. POST /chunks requires an existing stakeholderId; SG0 seeds exactly one bootstrap stakeholder for dev/test/Docker-verification use. Full stakeholder registration is deferred to a future goal.
