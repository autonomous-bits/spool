Scope notes (not Meridian ambiguity — explicit phasing decisions, consistent with G01/G02's own scope deviations):

  • No generic branch-overlay graph read. No branch-overlay/UNION-based "active graph" resolver exists yet in this repo (confirmed: ChunkRepository has no label lookup, only findById). This goal adds only a narrow ChunkRepository.findByLabel(label, branchId) used solely to validate that an edge's endpoints exist in the same scope (branch-local, or both branchless) — not a general-purpose graph-read API. A full active-graph resolver is deferred to a future goal.
  • No "or mainline" resolution. This repo has no implemented promotion pipeline yet (chunks are always status: 'draft'), so there is no real promoted mainline graph to resolve against. Edge endpoint validation in this goal is branch-local only: both fromChunkLabel/toChunkLabel must resolve to chunks in the same branchId (or, if branchId is omitted, both must resolve to branchless draft chunks). This "no branchId" path is a temporary resolution against the current branchless-draft scope only (no draft-vs-promoted precedence logic) and must be revisited once a real promotion pipeline exists.
  • No supersede/deactivate command. Schema/domain model EdgeStatus (active | superseded | deactivated) and superseded_by_edge_id so the data model is supersede-ready per IDEA-38, but this goal only ever creates status: 'active' edges. Actual supersede/deactivate endpoints are a future goal.

Sub-goal detail

SG1 — domain: Edge entity + EdgeType/EdgeStatus vocab open

Depends on: none

Objective: Model Edge as a validated domain entity with its closed type/status vocabularies, per IDEA-36/IDEA-37/IDEA-38.

  • EdgeType is a closed 8-value enum: refines | depends-on | contradicts | derives-from | blocks | implements | constrains | feedback-on (matches the authoritative schema's CHECK constraint verbatim).
  • EdgeStatus is a closed 3-value enum: active | superseded | deactivated. This goal's write path only ever produces active (mirrors ChunkStatus/BranchStatus precedent from G01/G02: model the full vocab, only produce one value for now).
  • Edge entity fields (all camelCase in domain/API; snake_case is DB-column-only, per SG2): id, fromChunkLabel, toChunkLabel, type: EdgeType, status: EdgeStatus (defaults 'active'), discipline: Discipline (reuses existing vocab), branchId?, originBranchId?, supersededByEdgeId? (always undefined this goal — reserved for a future supersede goal), createdByStakeholderId, updatedByStakeholderId, createdAt, updatedAt.
  • Constructor rejects: blank fromChunkLabel/toChunkLabel, fromChunkLabel === toChunkLabel (self-referential edges are invalid), invalid type, invalid discipline, blank createdByStakeholderId — by throwing, mirroring Chunk/Branch constructor style.
  • On initial creation, updatedByStakeholderId defaults to the same value as createdByStakeholderId (the incoming stakeholderId) — mirroring Chunk's existing creation behavior; there is no separate "updater" concept until a future edit/supersede goal.
  • Unit tests cover valid construction and each rejection case.

Verification: pnpm test:store (unit tests for the new domain module).

SG2 — persistence/infra: edges schema + EdgeRepository + label lookup open

Depends on: SG1

Objective: Add the authoritative edges table and the minimal persistence needed to create and read edges, plus a narrow chunk-label-existence check.

  • Migration creates edges(id, from_chunk_label, to_chunk_label, type, status, discipline, branch_id, origin_branch_id, superseded_by_edge_id, created_by_stakeholder_id, updated_by_stakeholder_id, created_at, updated_at) exactly per the authoritative schema (type/status CHECK constraints matching SG1's vocab; branch_id REFERENCES branches(id) ON DELETE CASCADE; origin_branch_id REFERENCES branches(id); superseded_by_edge_id REFERENCES edges(id), nullable, unused this goal).
  • Indexes created per the authoritative schema: idx_edges_branch_lookup (branch_id, from_chunk_label, to_chunk_label); idx_edges_mainline unique on (from_chunk_label, to_chunk_label, type) WHERE branch_id IS NULL AND status = 'active'; idx_edges_branch_active unique on (from_chunk_label, to_chunk_label, type, branch_id) WHERE status = 'active' — these two unique indexes are what enforce the "at most one active edge of a given type between two chunks" invariant (IDEA-38) at the database level.
  • EdgeRepository (mirrors ChunkRepository/BranchRepository pattern, parameterized queries only): create(edge): Promise<Edge>, findById(id): Promise<Edge | undefined>. No generic list/overlay-read method (out of scope per the goal-level scope note).
  • ChunkRepository gains findByLabel(label: string, branchId: string | undefined): Promise<Chunk | undefined> — looks up a chunk by label scoped to the given branchId (or branch_id IS NULL when branchId is undefined). Used only for edge-endpoint-existence validation in SG3, not exposed over any route in this goal.
  • Integration tests (containerized Postgres, per repo convention) cover: migration applies cleanly; EdgeRepository.create/findById round-trip; the two unique active-edge indexes reject a duplicate active edge (same from/to/type/branch) at the DB level; ChunkRepository.findByLabel resolves branch-scoped and branchless chunks correctly and returns undefined when not found.

Verification: pnpm test:store (migration + repository e2e tests against containerized Postgres).

SG3 — API: Edges controller/service open

Depends on: SG2

Objective: Expose POST /edges and GET /edges/:id, enforcing branch-local endpoint-existence and the single-active-edge invariant.

  • POST /edges accepts { fromChunkLabel, toChunkLabel, type, discipline, stakeholderId, branchId? } (mirrors CreateChunkRequest's shape/convention: discipline is an explicit required input, not derived, exactly as chunk creation already works).
  • Returns 400 for missing/blank fromChunkLabel/toChunkLabel/stakeholderId, invalid type, invalid discipline, or fromChunkLabel === toChunkLabel — mirroring parseCreateChunkRequest's IO-boundary-guard style.
  • When branchId is present: 404 if the branch does not exist; 409 if branch.discipline !== request.discipline (mirrors chunk-authoring's discipline-bound check from G02 SG3); 409 if branch.status !== 'draft' (write-lock, same as G02 SG3).
  • Endpoint existence check (branch-local only, per goal-level scope note): both fromChunkLabel and toChunkLabel must resolve via ChunkRepository.findByLabel(label, branchId) to an existing chunk in the same scope (same branchId, or both branchless when branchId is omitted); 404 if either does not resolve.
  • Single-active-edge enforcement: a duplicate active edge (same fromChunkLabel/toChunkLabel/type/branchId scope) is rejected with 409, by catching the unique-index violation from SG2 (same pattern as isForeignKeyViolation in chunks.service.ts, adapted for the unique-violation Postgres error code).
  • GET /edges/:id returns the created edge's full shape (id, fromChunkLabel, toChunkLabel, type, status, discipline, branchId, originBranchId, supersededByEdgeId, createdByStakeholderId, updatedByStakeholderId, createdAt, updatedAt) — supersededByEdgeId is always null this goal but is included now so the response shape doesn't drift when a future supersede goal populates it; 404 if not found.
  • e2e tests cover: successful branch-scoped creation, successful branchless creation, each 400/404/409 case above.

Verification: pnpm test:store (controller/service unit tests + new e2e spec against containerized Postgres).

SG4 — MCP exposure open

Depends on: SG3

Objective: Let an ideation assistant create a typed edge between two existing chunks, per IDEA-9.

  • New create-edge MCP tool mirrors create-branch's conventions exactly: validates untrusted input shape only (fromChunkLabel/toChunkLabel/type/discipline/stakeholderId as non-empty strings, branchId optional non-empty string), forwards to POST /edges, never invents a stakeholder identity, surfaces the store's 4xx errors as a typed validation error (e.g. CreateEdgeValidationError).
  • Vocabulary/invariant validation (closed type set, discipline match, endpoint existence, single-active-edge) remains the store's responsibility — not re-validated in the tool, per the existing tools' documented convention.
  • Unit tests cover: valid input parsing, missing-field rejection, and successful/error forwarding (mocked fetch) — same test shape as create-branch.spec.ts.

Verification: pnpm test:mcp.

SG5 — Docker end-to-end verification open

Depends on: SG4

Objective: Prove the full four-layer slice (MCP → API → domain → persistence) works against the real containerized system, not just in-process tests.

  • docker compose up --build spoolstore brings up the store against real Postgres.
  • Over real HTTP: create a branch (POST /branches), create two chunks scoped to that branch (POST /chunks with branchId), then POST /edges referencing both chunks' labels and that branchId succeeds; GET /edges/:id reflects the created edge.
  • A negative case is exercised over HTTP: repeating the same POST /edges (same from/to/type/branch) is rejected with 409 (single-active-edge invariant).
  • A second negative case is exercised over HTTP: POST /edges referencing a toChunkLabel that does not exist in that branch's scope is rejected with 404.
  • The MCP layer is exercised too: with spoolstore running via compose, separately start apps/mcp (e.g. node apps/mcp/dist/main.js) configured with HARNESS_URL pointed at the running store; invoke the create-edge tool to create an edge between two chunks created via the existing capture-chunk tool, confirmed via HTTP readback (GET /edges/:id) against the store. Both HTTP and MCP paths must be proven — not "either/or".

Verification: manual/scripted HTTP exercise (curl or equivalent) against spoolstore plus an MCP tool invocation (via a running apps/mcp process pointed at that same container, per above) with HTTP readback (not satisfied by unit/in-process tests alone).
