G05 — Branch Verification (submitted → verified | draft)

Status: open · Size: 4/5 — touches domain, persistence, and API layers; widens the existing ActorContext type (nullable discipline) to correctly model verification as a cross-discipline "merging authority" action distinct from discipline-scoped submission; adds a schema column and reject-path reset semantics; extends the read model. Reuses G04's GitHub OAuth session-token auth as-is (no new auth code), which keeps this below G04's 5/5.

Sub-goal detail

SG0 — domain: nullable ActorContext.discipline + verify/reject lifecycle assertions open

Depends on: none

Objective: Model verification/rejection as human-only, discipline-agnostic transitions, per IDEA-11/IDEA-25/IDEA-43, without weakening submit's existing discipline check.

  • Widen BaseActorContext.discipline in domain/types/actor/actor-context.ts from Discipline to Discipline | null. assertSubmitDiscipline's existing comparison (actor.discipline !== branch.discipline) is unaffected: BranchesService.submit already pre-guards on isDiscipline(stakeholder.discipline) before constructing the actor, so submit's behavior is unchanged.
  • Add assertSubmittedStatus(branch: Pick<Branch, 'status'>): void — throws a BranchLifecycleError unless branch.status === 'submitted' (verify's guard).
  • Add assertRejectableStatus(branch: Pick<Branch, 'status'>): void — throws a BranchLifecycleError unless branch.status === 'submitted' || branch.status === 'verified' (reject's guard).
  • Reuse assertIsHumanActor unchanged for both verify and reject. Do not add any discipline-match assertion for verify/reject — this discipline-agnostic design was confirmed by the user (see Open Questions: resolved).
  • Unit tests: each new assertion's pass/throw cases; a null-discipline actor passes assertIsHumanActor and is accepted by verify/reject call sites (no discipline check invoked at all for these two operations).

Verification: pnpm test:store (domain unit tests).

SG1 — persistence: verified_at column + BranchRepository.verify/reject open

Depends on: none

Objective: Persist the verify/reject transitions atomically, mirroring the existing submit() compare-and-set pattern, with reject fully resetting review state.

  • New migration: ALTER TABLE branches ADD COLUMN verified_at TIMESTAMP WITH TIME ZONE; (nullable, mirrors submitted_at's existing shape; no "verified-by" column, matching the repo's existing submitted_at-without-"submitted-by" precedent).
  • Extend Branch domain entity + BranchProps + BranchRepository's BranchRow/toBranch mapping with an optional verifiedAt: Date.
  • BranchRepository.verify(branchId): Promise<Branch | undefined> — transactional: BEGIN; SELECT ... FOR UPDATE the branch row plus a clock_timestamp() guard (identical structure to submit()); if the row is missing or status !== 'submitted', ROLLBACK and return undefined; otherwise UPDATE branches SET status='verified', verified_at=now(), updated_at=now() WHERE id=$1 RETURNING *, then COMMIT.
  • BranchRepository.reject(branchId): Promise<Branch | undefined> — same transactional shape: SELECT ... FOR UPDATE the branch row; if missing or status is not 'submitted'/'verified', ROLLBACK and return undefined; otherwise UPDATE branches SET status='draft', verified_at=NULL, submitted_at=NULL, updated_at=now() WHERE id=$1 RETURNING *, then COMMIT.
  • Both methods return undefined for "branch missing" and "branch has wrong status" alike (matching submit()'s existing shape) — SG2's service layer is responsible for telling these apart via its own prior findById lookup (404) before calling verify/reject (409 on an undefined result, since findById already proved the branch exists).
  • Integration tests: verify succeeds once from submitted and returns undefined on repeat; reject succeeds from both submitted and verified, clearing both verified_at and submitted_at; reject returns undefined from draft or merged.

Verification: pnpm test:store (persistence integration tests).

SG2 — API: POST /branches/:id/verify + POST /branches/:id/reject open

Depends on: SG0, SG1

Objective: Expose both transitions over HTTP, gated by the existing session-token bearer auth, with no discipline requirement, and surface verifiedAt on the read model.

  • BranchResponse gains a nullable verifiedAt, populated on successful verify and on GET /branches/:id; cleared (absent/null) after a reject.
  • BranchesService.verify(branchId, claims): findById(branchId) first — 404 (NotFoundException) if absent. Then resolves the stakeholder by claims.stakeholderId (400 if not found — no discipline-null rejection, unlike submit); constructs a HumanActorContext with whatever discipline the stakeholder has (including null); asserts human actor + assertSubmittedStatus (409/ConflictException on the found-but-wrong-status in-memory check, as a fast path). Calls BranchRepository.verify; 409 (ConflictException) if it returns undefined (covers the race where status changed between the read and the write).
  • BranchesService.reject(branchId, claims): identical control flow — findById first (404 if absent), same actor resolution, assertRejectableStatus fast-path 409, then BranchRepository.reject with a 409 on undefined.
  • BranchesController: POST :id/verify and POST :id/reject, both requiring Authorization: ****** via the existing verifySessionClaims helper (401 on missing/invalid/expired token, identical to submit).
  • e2e tests cover: verify success path (any discipline, including a stakeholder whose discipline differs from the branch's, and a null-discipline stakeholder); verify 409 from non-submitted status; reject success from both submitted and verified; reject 409 from draft/merged; 401 with no/invalid token on both endpoints; GET /branches/:id reflects verifiedAt correctly after each transition.

Verification: pnpm test:store (e2e HTTP tests).

SG3 — Docker end-to-end open

Depends on: SG2

Objective: Exercise verify and reject against the real containerized store over HTTP, confirming behavior matches the Meridian acceptance criteria.

  • docker compose up --build spoolstore against real Postgres, obtaining a real session token by running pnpm dev:session-token (tools/dev-session-token/get-dev-session-token.mjs) against the running container — this drives the real GET /auth/github/login → GET /auth/github/callback flow through the github-oauth-stub service already wired into compose.yaml. Do not fabricate, hand-sign, or otherwise bypass the token.
  • Real HTTP flow: create branch → submit → POST /branches/:id/verify with a token whose stakeholder discipline differs from the branch's discipline succeeds (proves discipline-agnosticism); GET /branches/:id shows status: 'verified' and a non-null verifiedAt.
  • Separate run: create branch → submit → POST /branches/:id/reject succeeds; GET /branches/:id shows status: 'draft', verifiedAt and submittedAt both null/absent.
  • Repeat verify on an already-verified branch returns 409; verify with no Authorization header returns 401.

Verification: manual/scripted HTTP exercise against spoolstore covering the success paths (verify, reject) and the 409/401 negative cases.

Open Questions

None. The prior open question — whether verify/reject should be discipline-agnostic (no actor/branch discipline match, unlike submit) — is resolved: confirmed by the user (repo maintainer) on 2026-07-06 that any human stakeholder, regardless of discipline, may verify/reject a branch. This matches the reading already inferred from IDEA-11's "merging authorities" language vs. IDEA-35/IDEA-53's submission-only discipline-scoping wording. SG0/SG2 are unchanged by this confirmation (they already implement it this way); no ratified Meridian chunk contradicts it, so no Meridian escalation is needed.
