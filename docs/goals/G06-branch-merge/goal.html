Sub-goal Detail

SG0 — Domain: merge lifecycle guard + lineage/provenance types
Layer: domain  |  Depends on: none

Objective: Add the domain-level guard and types needed before any persistence work can safely promote branch content to mainline.

  • Add assertMergeableStatus(branch) to apps/store/src/domain/branch-lifecycle.ts: throws BranchLifecycleError unless branch.status === 'verified' (IDEA-43). Reuse existing assertIsHumanActor for the human-only check (IDEA-42/IDEA-57) — no new human-check code.
  • Add mergedAt?: Date and mergedByStakeholderId?: string to Branch/BranchProps in apps/store/src/domain/branch.ts.
  • Add a new MergeLineage type (new file, e.g. apps/store/src/domain/merge-lineage.ts) per IDEA-74: { branchId, discipline, divergedAt: DivergencePoint, mergedAt: Date, mergedByStakeholderId }. Explicitly omit workspaceId — no workspace concept exists anywhere in this repo yet (verified: no workspace_id column or type in apps/store/src); document this scope deviation in the file's doc comment, mirroring the existing origin_suggestion_id precedent in migration 0003.
  • Add a new BranchGraphProvenance type (new file, e.g. apps/store/src/domain/branch-graph-provenance.ts) per IDEA-74, scoped to { sourceBranchId } only — a thin wrapper documenting the existing originBranchId field's semantic role. Explicitly omit sourceWorkspaceId/sourceDiscipline for the same reason as above; document the deviation.

Acceptance criteria:

  • assertMergeableStatus throws for every non-'verified' status and passes for 'verified'.
  • Branch round-trips mergedAt/mergedByStakeholderId when provided, and leaves them undefined when omitted (matches existing submittedAt/verifiedAt pattern).
  • MergeLineage and BranchGraphProvenance construct successfully with valid inputs and are covered by unit tests; both carry a doc comment stating the workspaceId/sourceWorkspaceId/sourceDiscipline omission and citing IDEA-74.

Verification: pnpm test:store (targeted: branch-lifecycle.spec.ts, branch.spec.ts, new merge-lineage.spec.ts, new branch-graph-provenance.spec.ts)

SG1 — Persistence: atomic conflict-checked merge transaction
Layer: persistence/infra  |  Depends on: SG0

Objective: Implement BranchRepository.merge(branchId, mergedByStakeholderId) as a single atomic transaction that conflict-checks then promotes a verified branch's chunks/edges into the mainline.

  • Row-lock the branch (SELECT ... FOR UPDATE, matching submit/verify/reject); if not found or status !== 'verified', rollback and return undefined (mirrors existing repository return-undefined-for-conflict convention).
  • Conflict check (concrete scope for IDEA-46/IDEA-54, given no chunk/edge update-or-override mechanism exists yet in this repo): for every chunk in the branch (branch_id = branchId), check whether a mainline chunk already exists with the same label where branch_id IS NULL AND status = 'promoted' (this is exactly the row-set guarded by the existing idx_chunks_mainline partial unique index). For every edge in the branch, check whether a mainline edge already exists with the same (from_chunk_label, to_chunk_label, type) where branch_id IS NULL AND status = 'active' (guarded by idx_edges_mainline). Any single collision aborts the entire merge (rollback, no partial promotion) and the method returns a distinct conflict marker (e.g. a discriminated result, not undefined, so the service layer can tell "conflict" apart from "not mergeable status").
  • If no conflicts: UPDATE chunks SET branch_id = NULL, status = 'promoted', updated_at = now() WHERE branch_id = $branchId; UPDATE edges SET branch_id = NULL, updated_at = now() WHERE branch_id = $branchId (edge status stays 'active'; origin_branch_id is never modified on either table — it already equals the branch id from creation and must persist permanently per IDEA-69).
  • UPDATE branches SET status = 'merged', merged_at = now(), merged_by_stakeholder_id = $mergedByStakeholderId, updated_at = now() WHERE id = $branchId RETURNING * (columns already exist in schema since migration 0003 — no new migration needed).
  • Commit; return the merged Branch. On any exception mid-transaction, rollback fully (IDEA-47).

Acceptance criteria:

  • Merging a verified branch with no mainline collisions promotes every one of its chunks (now branch_id IS NULL, status = 'promoted', origin_branch_id unchanged) and edges (branch_id IS NULL, origin_branch_id unchanged) and sets the branch to status = 'merged' with merged_at/merged_by_stakeholder_id populated — all within one transaction.
  • A branch with a chunk-label or edge-identity collision against mainline is rejected in full (zero chunks/edges promoted, branch status unchanged) and a distinguishable conflict result is returned.
  • A branch not in 'verified' status returns undefined (mirrors submit/verify/reject convention) without mutating any row.
  • A simulated mid-transaction failure (e.g. a forced error after the chunk UPDATE but before the branch UPDATE) leaves the database in its pre-merge state (full rollback, IDEA-47).

Verification: pnpm test:store (targeted: new branch.repository.e2e-spec.ts merge cases, run against real containerized Postgres per the existing repository e2e pattern)

SG2 — API: POST /branches/:id/merge
Layer: API/URL surface  |  Depends on: SG1

Objective: Expose branch merge over HTTP, reusing the exact auth/discipline-agnostic pattern already established by verify/reject.

  • BranchesController: add @Post(':id/merge') calling verifySessionClaims (identical to submit/verify/reject) then branches.merge(id, claims).
  • BranchesService.merge(branchId, claims): reuse resolveActorForVerification (discipline-agnostic, human-only — same as verify/reject per G05's resolved question) then assertIsHumanActor + assertMergeableStatus; call branchRepository.merge(...); map "not mergeable" → ConflictException, "mainline collision" → ConflictException with a distinct message, success → toBranchResponse.
  • BranchResponse DTO: add mergedAt field (nullable, matching existing verifiedAt pattern).

Acceptance criteria:

  • Merging a verified branch via a valid Bearer session token returns 200 with mergedAt populated in the response body.
  • Merging a non-verified branch (draft/submitted/already-merged) returns 409.
  • Merging a branch with a mainline identity collision returns 409 with a message distinguishing it from the non-verified-status 409.
  • Missing/invalid Authorization header returns 401 (reusing existing extractBearerToken/verifySessionClaims — no new code needed here).
  • Merge is discipline-agnostic: a human stakeholder from a different discipline than the branch's authoring discipline can still merge it (matches G05's resolved "merging authority" interpretation of IDEA-11).

Verification: pnpm test:store (targeted: branches.controller.spec.ts, branches.service.spec.ts, test/branches.e2e-spec.ts)

SG3 — Docker end-to-end exercise (mandatory)
Layer: full stack, containerized  |  Depends on: SG2

Objective: Prove the full merge flow works against the real running system, not just in-process tests. This step is not satisfied by unit/integration tests alone.

  • Bring the system up with docker compose up --build spoolstore (per apps/store/AGENTS.md).
  • Obtain a real session token via pnpm dev:session-token (drives the actual GET /auth/github/login → GET /auth/github/callback flow against the running container, using the wired-in github-oauth-stub service). Do not substitute a hand-crafted or client-supplied token — this is the only faithful way to exercise the human-only merge gate.
  • Part 1 (HTTP, success path): over real HTTP against the running container — create a branch, capture a chunk and an edge into it, submit, verify, then POST /branches/:id/merge with the real token. Confirm 200 and a populated mergedAt in the JSON response.
  • Part 2 (SQL, persistence invariants): because the HTTP DTOs do not expose branch_id/origin_branch_id/chunk status, connect directly to the running containerized Postgres and confirm: the merged chunk has branch_id IS NULL, status = 'promoted', origin_branch_id still equal to the original branch id; the merged edge has branch_id IS NULL and the same origin_branch_id; the branch row has status = 'merged', non-null merged_at and merged_by_stakeholder_id.
  • Part 3 (conflict path): repeat the flow with a second branch authoring a chunk with the same label already promoted to mainline by Part 1; verify it; attempt merge; confirm HTTP 409 and, via SQL, that none of its chunks/edges/branch row changed.

Acceptance criteria: All three parts pass against the real containerized store; the success path's mergedAt is visible over HTTP; the persistence invariants (Part 2) and conflict rejection (Part 3) are confirmed via direct SQL against the running Postgres container, not via unit/integration tests.

Verification: Manual/scripted exercise per above (no single pnpm command — this is the Docker end-to-end step required by every goal).

Open Questions

Both resolved 2026-07-06 with @pewpewpotato — proceed as scoped, no Meridian escalation for either.

  • [RESOLVED] MergeLineage/BranchGraphProvenance workspaceId scope (SG0): IDEA-74 names workspaceId on MergeLineage and sourceWorkspaceId/sourceDiscipline on BranchGraphProvenance, but no workspace concept exists anywhere in this repo's domain or schema today. Decision: implement both types scoped down to only the fields this repo can actually persist (branchId/discipline/divergedAt/mergedAt/mergedByStakeholderId for MergeLineage; sourceBranchId only for BranchGraphProvenance), deferring the workspace-scoped fields to whichever future goal introduces a workspace concept. This is a repo-local scoping decision (not a Meridian conflict) and mirrors existing precedent (e.g. origin_suggestion_id declared without an FK in migration 0003 because the suggestions table doesn't exist yet).
  • [RESOLVED] Conflict-detection scope (SG1): IDEA-46 describes detecting modifications made in both branch and mainline since divergence. Because no chunk/edge update-or-supersede mechanism exists yet in this repo, this goal implements the closest buildable, schema-enforced equivalent: any mainline identity collision (same chunk label, or same edge from/to/type) blocks the merge outright, regardless of exact timing. This is stricter than "since divergence" (rejects same-identity mainline writes made even before divergence) but is the only conflict signal the current schema can express, and it prevents the unique-index violations that a naive promotion would otherwise hit. Not escalated to Meridian as a conflict — it is a superset-safe, repo-local scoping decision pending a future versioning/override goal.
