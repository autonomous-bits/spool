Locked Design Decisions (not ambiguous, but must not be re-derived or re-guessed)

  • branchId scope = strict, not overlay. Prior goals (G03/G04) explicitly deferred generic branch-overlay/UNION reads (edge.repository.ts: "Meridian IDEA-33's branch-overlay UNION read is out of scope until a future goal"; chunk.repository.ts findByLabel: "not a generic branch-overlay/UNION read"). This goal continues that precedent: when branchId is supplied, both GET /chunks and the neighbourhood traversal match only rows whose own branch_id = branchId (plus, for neighbourhood, the root chunk itself regardless of its branch) — never a merged view of branch-scoped rows unioned with mainline rows. When branchId is omitted, only branchless/mainline-equivalent rows are visible (no discipline gate applies, per IDEA-119).
  • Cursor ordering/tie-break (IDEA-113). With q supplied: order by ts_rank DESC, created_at DESC, id DESC. Without q: order by created_at DESC, id DESC. The cursor encodes the last row's tie-break fields (opaque, e.g. base64 JSON) so pages are stable even with duplicate created_at values.
  • Neighbourhood dedupe/cycles (IDEA-112). Traversal uses a visited-chunk-id set; a chunk reachable via multiple paths/hops appears exactly once in the response, recorded at its minimum hop distance. Depth > 5 → 400 Bad Request.
  • Auth pattern reuse. Do not add a new guard/interceptor framework. Reuse the existing verifySessionClaims(authorizationHeader, sessionTokenService) (apps/store/src/auth/session-claims.helper.ts) and assertWorkspaceScope(headerWorkspaceId, { tier: 'token', workspaceIdClaim }) (apps/store/src/domain/workspace-scope.ts) exactly as branches/notifications controllers already do.
  • MCP tools require both fields. Per IDEA-125, search-chunks/get-neighbourhood take a required sessionToken field (forwarded as Authorization header in the same format extractBearerToken parses today, and, per IDEA-121/G11 precedent (e.g. capture-chunk.ts), a required workspaceId field (forwarded as X-Workspace-Id). Missing/blank either field → the MCP tool fails fast with a 401/400-equivalent tool error before calling the store.

Sub-goal Detail

SG1 — Chunk Search vertical slice done

Layer: persistence + domain + API + MCP (full vertical slice)  |  Depends on: none

Objective: Implement GET /chunks search/filter/pagination end to end, plus the search-chunks MCP tool, both session-token gated.

  • Acceptance criteria:
    • ChunkRepository gains a search(...) method supporting combinable AND filters: discipline, chunkType, status, contextKind, branchId (strict-scope per Locked Design Decisions), q (Postgres full-text ranked search across label+content, e.g. to_tsvector/plainto_tsquery/ts_rank), workspace-scoped always.
    • Cursor pagination: default limit 20, max 100; a caller-supplied limit > 100 is clamped down to 100 (not rejected with 400), consistent with IDEA-113's framing as a server-enforced cap; opaque cursor token, ordering/tie-break exactly as specified in Locked Design Decisions.
    • GET /chunks controller route added to ChunksController: requires an Authorization header in the same format extractBearerToken parses today (via verifySessionClaims) and X-Workspace-Id matching the token's workspaceId claim (via assertWorkspaceScope tier 'token') — no delegated/tokenless mode, per IDEA-115.
    • If branchId supplied and token's discipline claim is null → 400. If resolved but mismatched vs. the branch's discipline → 403. If branchId omitted → no discipline gate (mainline visible cross-discipline).
    • Response shape: { chunks: ChunkResponse[], nextCursor: string | null }.
    • New MCP tool apps/mcp/src/tools/search-chunks.ts: required inputs include sessionToken and workspaceId (plus the same filter params); forwards Authorization and X-Workspace-Id headers to the store; fails fast (no store call) if either is missing/blank.
    • Unit tests for the repository query building/tie-break and the discipline-gate helper; e2e tests in apps/store/test covering: no filters, each filter individually, combined filters, pagination across >1 page, missing token (401), missing workspace header (403), branch discipline mismatch (403), branch with no-discipline token (400).
    • MCP integration test for search-chunks covering the missing-sessionToken and missing-workspaceId fail-fast paths plus a successful forwarded call.

Verification: pnpm test:store, pnpm test:mcp

SG2 — Neighbourhood Retrieval vertical slice open

Layer: persistence + domain + API + MCP (full vertical slice)  |  Depends on: SG1

Objective: Implement GET /chunks/:id/neighbourhood multi-hop traversal end to end, plus the get-neighbourhood MCP tool, reusing SG1's session-token auth and discipline-gate helper.

  • Acceptance criteria:
    • Traversal query (edge/chunk repositories) walks outgoing+incoming edges from the root chunk up to depth hops (default 1, max 5; depth > 5 → 400).
    • Visited-set dedupe: a chunk reachable via multiple paths appears once, at its minimum hop distance (per Locked Design Decisions).
    • If branchId supplied: strict branch-scope (per Locked Design Decisions) applied at every hop, not just the root; discipline gate (400 no-discipline / 403 mismatch, identical semantics to SG1) re-checked at every hop — a multi-hop walk must never surface a different-discipline branch's chunks at any hop.
    • GET /chunks/:id/neighbourhood controller route added to ChunksController, same auth requirements as SG1 (verifySessionClaims + assertWorkspaceScope tier 'token').
    • Response shape: { chunk: ChunkResponse, neighbours: { edgeId, chunkId, label, content, type, status, discipline, contextKind, direction, hop }[] }.
    • New MCP tool apps/mcp/src/tools/get-neighbourhood.ts: required sessionToken + workspaceId inputs, same forwarding/fail-fast behavior as SG1's search-chunks.
    • Unit tests for traversal dedupe/cycle-safety and per-hop discipline gating; e2e tests in apps/store/test covering: depth 1 default, depth > 1 multi-hop, depth > 5 (400), a graph cycle (root reachable back to itself), branch-scoped traversal with matching/mismatched/no discipline.
    • MCP integration test for get-neighbourhood mirroring SG1's fail-fast coverage.

Verification: pnpm test:store, pnpm test:mcp

SG3 — Docker end-to-end verification open

Layer: Docker / system  |  Depends on: SG2

Objective: Exercise both new routes against the real containerized store using a real session token.

  • Acceptance criteria:
    • Build and run with docker compose up --build spoolstore (per apps/store/AGENTS.md); run pnpm build/pnpm --filter store build first so the container runs current code.
    • Obtain a real session token via pnpm dev:session-token (drives the real GET /auth/github/login → GET /auth/github/callback flow against the running container via the github-oauth-stub service already wired into compose.yaml). Do not substitute a hand-crafted/manually-signed token or a client-supplied field — this is the only faithful way to exercise every auth-gated scenario below.
    • Using that token: call GET /chunks with at least one filter and confirm a 200 with the expected { chunks, nextCursor } shape against real seeded/created data.
    • Call GET /chunks/:id/neighbourhood on a chunk with at least one edge and confirm a 200 with the expected { chunk, neighbours } shape, including a hop field.
    • Negative cases against the running container: omit the Authorization header (401), omit X-Workspace-Id (403), and (if a second-discipline branch/stakeholder fixture is available) a cross-discipline branch-filtered query (403).
    • Repeat the successful search/neighbourhood calls via the MCP server's search-chunks/get-neighbourhood tools (built from apps/mcp), confirming the token/workspace forwarding works end to end through the MCP layer, not just in-process.

Verification: Manual/tooled exercise of the running Docker container as described above; no unit/in-process test satisfies this step.
