G12 — Downstream Delivery Service (Webhooks) open

Size: 5/5 — two new domain concepts (DeliverySubscription, DeliveryAttempt), a new Postgres schema (two tables), a cross-layer contract spanning API + a durable background worker + HMAC webhook signing + branch-merge integration, and no MCP layer (see resolution below).

Scope framing: this goal delivers Push delivery via webhooks only (IDEA-63's "registered downstream containers, external webhooks, and egress lanes" — this slice covers webhooks; other channels are future work). Active querying (Pull) already exists via existing GET endpoints on chunks/edges/branches/artifacts and is unaffected by this goal.
Escalation resolved before this goal was written: IDEA-104 point 4 originally required 3 MCP tools (register/list/remove) for delivery-subscription CRUD, but also marked that same action "Human-only... no delegated/AI agent registration" — contradicting every other human-only precedent in this repo (branch submit/verify/merge, notification inbox, workspace registry, all ship with zero MCP tool exposure). This was escalated as IDEA-108 (conflict report — meridian-submit-feedback failed with a persistent 500, so it was captured via meridian-capture-draft-chunk instead, matching the IDEA-70/71/86 fallback pattern). A human architect confirmed the resolution directly in this session: no MCP tool for delivery-subscription CRUD, matching precedent. This is recorded as IDEA-109 (resolution chunk). Both IDEA-108 and IDEA-109 have since been approved and promoted to implementation-context — this decision is now ratified, not merely a draft escalation.

Sub-goal Detail

SG1 — Domain + persistence open

Layer: Domain / Persistence  |  Depends on: none

Objective: Add DeliverySubscription and DeliveryAttempt domain types (apps/store/src/domain/delivery-subscription.ts, delivery-attempt.ts) plus migration 0014_add_delivery_subscriptions_tables.sql and repositories, following existing domain/persistence conventions (see artifact.ts / artifact.repository.ts for the closest-shaped precedent: workspace-scoped, immutable-once-created secret field).

Exact schema for delivery_subscriptions (authoritative, per IDEA-104 point 3 — do not deviate):

CREATE TABLE delivery_subscriptions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id UUID NOT NULL REFERENCES workspaces(id),
  url TEXT NOT NULL,
  discipline_filter JSONB NULL,
  signing_secret TEXT NOT NULL,
  is_active BOOLEAN NOT NULL DEFAULT true,
  created_by_stakeholder_id UUID NOT NULL REFERENCES stakeholders(id),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

delivery_attempts schema (repo-owned engineering detail, not separately Meridian-ratified — required to satisfy IDEA-104's retry/FIFO/idempotency invariants durably, per rubber-duck review; do not implement as in-memory/ephemeral state):

CREATE TABLE delivery_attempts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  subscription_id UUID NOT NULL REFERENCES delivery_subscriptions(id),
  idempotency_key TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending', -- pending | succeeded | failed
  attempt_count INT NOT NULL DEFAULT 0,
  next_attempt_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  last_error TEXT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (subscription_id, idempotency_key)
);

Acceptance criteria:

  • assertValidWebhookUrl(url) throws unless url parses and starts with https:// (IDEA-104 pt.2).
  • normalizeDisciplineFilter(input?: string[]) returns null for omitted/empty input, meaning "all disciplines" (IDEA-104 pt.2); non-empty arrays pass through validated against the existing Discipline vocabulary type.
  • matchesDiscipline(subscription, discipline) returns true when disciplineFilter === null or it includes the given discipline.
  • Signing secret is generated with node:crypto randomBytes(32).toString('hex') at creation time and never regenerated (immutable, mirrors artifact-blob immutability precedent).
  • DeliverySubscriptionRepository: create (workspace-scoped), list (workspace-scoped, active only by default), deactivate (soft-delete via is_active = false, not a hard delete — mirrors branch/suggestion soft-lifecycle precedent), and a findActiveForWorkspaceAndDiscipline(workspaceId, discipline) lookup used by SG3.
  • DeliveryAttemptRepository: enqueue(subscriptionId, idempotencyKey) using ON CONFLICT (subscription_id, idempotency_key) DO NOTHING for idempotency; claimNextBatch(limit) selecting pending rows with next_attempt_at <= now() ordered by created_at ascending per subscription (FIFO); markSucceeded(id); markFailedOrRetry(id, error) incrementing attempt_count and computing exponential backoff (next_attempt_at), transitioning to permanent failed after the 3rd failed attempt (IDEA-104 pt.2).
  • Unit tests cover all validators, the repository's idempotent enqueue (duplicate enqueue is a no-op), and the backoff/permanent-failure transition logic.

Verification: pnpm --filter store test -- delivery-subscription delivery-attempt

SG2 — API layer (subscription CRUD, human-only) open

Layer: API  |  Depends on: SG1

Objective: Add a DeliverySubscriptionsController (new module, following the WorkspacesController pattern exactly — session-token gated via verifySessionClaims, X-Workspace-Id header validated against the token's workspaceId claim) exposing POST/GET/DELETE /workspaces/:workspaceId/delivery-subscriptions[/:id]. No MCP tool is added for this action, per IDEA-109.

Acceptance criteria:

  • POST /workspaces/:workspaceId/delivery-subscriptions body {url, disciplineFilter?} → 201 {id, workspaceId, url, disciplineFilter, signingSecret, isActive, createdAt} — signingSecret is returned only on this create response (never on GET/list), since the caller needs it once to configure their receiver's signature verification.
  • GET /workspaces/:workspaceId/delivery-subscriptions → 200 list of subscriptions, omitting signingSecret from every list-response item.
  • DELETE /workspaces/:workspaceId/delivery-subscriptions/:id → 204, soft-deactivates (SG1's deactivate).
  • Invalid/non-https:// url → 400.
  • Missing/invalid session token on any of the 3 endpoints → 401.
  • Valid token but caller not a member of :workspaceId → 403 (mirrors existing workspace-membership enforcement pattern from G11).
  • Nonexistent :id on DELETE → 404.
  • Controller + e2e-spec tests cover every case above.

Verification: pnpm --filter store test -- delivery-subscriptions.controller

SG3 — Merge-integration enqueue open

Layer: Domain / Persistence (integration)  |  Depends on: SG1

Objective: In apps/store/src/branches/branches.service.ts's merge(), after a successful mainline commit, enqueue one delivery_attempts row per active subscription in the branch's workspace whose disciplineFilter matches the branch's discipline (SG1's matchesDiscipline). Idempotency key = the merging branch's own id — merge is a one-way terminal transition (a branch merges exactly once, per existing lifecycle invariants), so branchId alone uniquely identifies "this merge event," with no separate merge-event-id concept needed.

Acceptance criteria:

  • On successful merge, exactly one delivery_attempts row is enqueued per matching active subscription, in the same transaction as the merge (or immediately after, if transactional coupling is impractical — must not be lost if the process crashes between the two; prefer same-transaction).
  • Subscriptions with disciplineFilter not matching the branch's discipline are skipped.
  • Inactive subscriptions (is_active = false) are skipped.
  • Enqueue is idempotent: re-attempting enqueue for the same branchId/subscription pair (e.g. from a retried code path) does not create a duplicate row (SG1's ON CONFLICT DO NOTHING).
  • Merge failure (conflict, already-merged, etc.) never enqueues any delivery attempt.
  • Unit/integration tests cover: matching-discipline enqueue, non-matching-discipline skip, inactive-subscription skip, idempotent re-enqueue, no-enqueue-on-merge-failure.

Verification: pnpm --filter store test -- branches.service merge-delivery

SG4 — Delivery worker (sign, POST, retry/backoff) open

Layer: Domain / Infra (background worker)  |  Depends on: SG1, SG3

Objective: Add a DeliveryWorkerService implementing IDEA-63's "background queue worker": on a scheduled poll interval (started in a NestJS lifecycle hook, e.g. onModuleInit, and stopped in onModuleDestroy per node-memory-management shutdown conventions), claims pending delivery_attempts (SG1's claimNextBatch, FIFO per subscription), builds a visibility-resolved payload for the merged branch, signs it, and POSTs it to the subscription's url.

Signing: add signWebhookPayload(bodyJson: string, secret: string): string to apps/store/src/auth/ (new function, not reusing hmac-token.ts's envelope-token codec since webhook signatures are a raw HMAC-SHA256 hex digest over the exact JSON body, not an enveloped/expiring token) — returns the hex digest, sent as an X-Spool-Signature: sha256=<hex> header alongside the raw JSON body.

Acceptance criteria:

  • Payload includes at minimum: branchId, discipline, workspaceId, mergedAt, mergedByStakeholderId (from MergeLineage).
  • 2xx response from the subscriber → markSucceeded.
  • Non-2xx or network error → markFailedOrRetry; retries use exponential backoff (e.g. 1s / 4s / 16s or equivalent doubling+ scheme) for up to 3 total attempts (IDEA-104 pt.2); after the 3rd failed attempt, status is permanently failed and no further retries are scheduled.
  • FIFO ordering: for a given subscription, an older pending attempt is always dispatched before a newer one; attempts across different subscriptions may interleave.
  • Worker poll loop starts on module init and is cleanly stopped on module destroy (no dangling timers/handles — verified per node-memory-management conventions).
  • Unit tests (HTTP client mocked) cover: successful delivery marks succeeded, failed delivery schedules backoff retry, 3rd failure marks permanently failed, FIFO ordering within one subscription, signature header matches an independently-computed HMAC-SHA256 digest of the exact body sent.

Verification: pnpm --filter store test -- delivery-worker

SG5 — Docker end-to-end exercise (mandatory, final) open

Layer: docker-e2e  |  Depends on: SG2, SG4

Objective: Exercise the full webhook delivery slice against the real running containerized store, using a real human session token and a real webhook receiver.

Acceptance criteria:

  • Add a minimal webhook-receiver stub service to compose.yaml (mirrors apps/store/docker/github-oauth-stub's pattern: a tiny server.mjs that records each received POST — headers, raw body — and exposes a simple GET to retrieve recorded calls for assertions).
  • Rebuild host dist/ (pnpm --filter store build) before docker compose up --build spoolstore, per this repo's Docker-staleness precedent.
  • Obtain a real human session token via pnpm dev:session-token (real OAuth login/callback flow against github-oauth-stub) — required because subscription CRUD is human-only/session-token gated; do not substitute a hand-crafted or manually-signed token.
  • Using that token: create/bootstrap a workspace if needed, register a delivery subscription pointing at the webhook-receiver stub's container URL, then run the full branch pipeline (create branch → author a chunk → submit → verify → merge) using the same token.
  • Confirm the webhook-receiver stub recorded exactly one POST for the merge, with a valid X-Spool-Signature header verifiable against the subscription's returned signingSecret, and a body referencing the merged branchId.
  • Confirm exactly one delivery_attempts row exists for that branch/subscription pair with status = succeeded (checked directly against the running Postgres container), demonstrating idempotency (no duplicate rows, no duplicate webhook POSTs).

Verification: manual Docker exercise (docker compose up --build spoolstore + pnpm dev:session-token + curl against the store and the webhook-receiver stub's inspection endpoint + direct Postgres query)
