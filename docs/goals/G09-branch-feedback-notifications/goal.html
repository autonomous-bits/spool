Sub-goal Detail

SG1 — submit verification signal done

Layer: domain + persistence + API   Depends on: none

Objective: Any named verifier (agent, tool, or human) can log a pass/fail evaluation against a branch under review; it is recorded as feedback but never auto-transitions the branch.

  • verification_signals table exactly matches schema.sql (ratified via IDEA-31, no deviation needed): id UUID PK, branch_id UUID FK branches(id) ON DELETE CASCADE, verifier_name VARCHAR(255) NOT NULL (free text — not a stakeholder FK, since IDEA-21 explicitly allows "agents, tools, or other humans" that need not be registered stakeholders), status VARCHAR(50) NOT NULL CHECK IN ('pass','fail'), reason TEXT (nullable), created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW().
  • Migration 0010_add_verification_signals_table.sql creates this table only (feedback_notifications ships in SG2's migration).
  • POST /branches/:id/verification-signals: unauthenticated (mirrors POST /suggestions precedent — no actor-context/session-token needed since verifier_name is free text, not an authenticated identity). Body: { verifierName: string, status: 'pass' | 'fail', reason?: string }.
  • Domain rule (IDEA-20/IDEA-43): signal submission is only allowed when the branch status is submitted or verified (the review window) — draft or merged branches reject with 409. Add assertReviewableStatus to branch-lifecycle.ts alongside the existing assert* helpers.
  • Submitting a signal never changes branches.status (IDEA-43 is explicit: no automated state transition) — the human stakeholder still calls the existing POST /branches/:id/verify/reject to decide.
  • GET /branches/:id/verification-signals: unauthenticated list read, ordered by created_at ascending, matching this codebase's existing unauthenticated-GET precedent (GET /branches/:id, GET /suggestions).

Acceptance criteria:

  • POST /branches/:id/verification-signals on a submitted or verified branch with a valid body persists one row and returns it; unknown branchId → 404.
  • Same request against a draft or merged branch → 409, no row persisted.
  • status outside 'pass'|'fail', or blank verifierName, → 400.
  • Submitting a signal does not change the branch's status field (assert via GET /branches/:id before/after).
  • GET /branches/:id/verification-signals returns all signals for that branch ordered oldest-first, with no Authorization header required.

Verification: pnpm test:store

SG2 — notification fan-out done

Layer: domain + persistence (extends SG1's endpoint)   Depends on: sg1-submit-verification-signal

Objective: Every verification signal immediately creates one unread notification per stakeholder in the system, so any of them (as a potential merging authority per IDEA-11) is promptly alerted.

  • feedback_notifications table exactly matches schema.sql (ratified via IDEA-31): id UUID PK, branch_id UUID NOT NULL FK branches(id) ON DELETE CASCADE, stakeholder_id UUID NOT NULL FK stakeholders(id) ON DELETE CASCADE, signal_id UUID NOT NULL FK verification_signals(id) ON DELETE CASCADE, status VARCHAR(50) NOT NULL CHECK IN ('unread','read'), created_at/updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(); index idx_feedback_notifications_stakeholder (stakeholder_id, status) per schema.sql.
  • Migration 0011_add_feedback_notifications_table.sql creates this table + index.
  • Documented recipient-scope decision (literal reading of IDEA-67 "the author of the branch and other stakeholders" applied to the only membership model that exists — a flat, global stakeholders table with no branch-watcher/subscription concept): fan-out targets every stakeholder in the system, not a discipline-scoped subset. This is consistent with G05's already-ratified rule that any human stakeholder, regardless of discipline, may act as a merging authority for the branch — so any of them is a legitimate notification recipient.
  • Add StakeholderRepository.findAll(): Promise<StakeholderRecord[]> (new method; the repository currently only exposes findById/findByGithubLogin).
  • SG1's POST /branches/:id/verification-signals handler becomes one atomic DB transaction: insert the verification_signals row, then insert one feedback_notifications row (status='unread') per row returned by findAll() — all-or-nothing, mirroring G07 SG2's accept-suggestion transaction pattern.

Acceptance criteria:

  • Submitting a valid verification signal when N stakeholders exist creates exactly N feedback_notifications rows, all status='unread', all referencing the new signal's id and the branch's id.
  • A stakeholder created after signal submission does not retroactively receive a notification for that earlier signal.
  • A failure injected mid-transaction (e.g. simulated notification insert failure) rolls back the signal row too — no orphaned signal without notifications.

Verification: pnpm test:store

SG3 — notification consumption done

Layer: domain + persistence + API (no MCP tool — personal inbox reads are human-only, mirrors G07's no-MCP-tool precedent for human-only ops)   Depends on: sg2-notification-fanout

Objective: A human stakeholder can list their own notifications and mark one as read.

  • New module: apps/store/src/notifications/ with NotificationsModule, NotificationsController, NotificationsService, plus a NotificationRepository in apps/store/src/persistence/ — mirrors the existing per-domain module layout (branches/, suggestions/, artifacts/).
  • Auth: extract the bearer-token-extraction and session-claims-verification logic currently duplicated inline in BranchesController and SuggestionsController (extractBearerToken/verifySessionClaims) into one shared helper (e.g. apps/store/src/auth/session-claims.helper.ts), and have NotificationsController use that shared helper — do not write a third inline copy.
  • GET /notifications: returns only the caller's own feedback_notifications rows (scoped by the session token's stakeholderId, never a client-supplied id), ordered by created_at descending. Optional ?status=unread query filter.
  • POST /notifications/:id/read: sets status='read', updated_at=NOW(); 404 if the notification does not exist OR does not belong to the caller (never leak existence of another stakeholder's notification via a 403); idempotent — marking an already-read notification read again returns 200 unchanged.
  • Missing/invalid/expired session token on either route → 401 (mirrors existing branch-lifecycle auth tests).

Acceptance criteria:

  • GET /notifications with a valid session token returns only that stakeholder's rows, newest first.
  • GET /notifications?status=unread returns only unread rows for that stakeholder.
  • POST /notifications/:id/read on the caller's own unread notification returns it with status='read'.
  • POST /notifications/:id/read on a notification seeded (directly via the repository in the test, not through Docker) as belonging to a different stakeholder → 404 (not 403 — do not reveal existence).
  • Both routes without a valid session token → 401.

Verification: pnpm test:store

SG4 — MCP submit-verification-signal done

Layer: MCP   Depends on: sg1-submit-verification-signal

Objective: A delegated actor (evaluation/ideation assistant, per IDEA-75's delegated-actor-eligible advisory feedback) can submit a verification signal via MCP, mirroring the existing submit-suggestion tool's structure.

  • New file apps/mcp/src/tools/submit-verification-signal.ts, structured exactly like submit-suggestion.ts: own parse*Input, *ValidationError, result type, spec file.
  • Input: { branchId: string, verifierName: string, status: 'pass' | 'fail', reason?: string }. Forwards unchanged to the store's POST /branches/:id/verification-signals (SG1). Never re-validates the status vocabulary itself — the store is authoritative and its 4xx errors must be surfaced, not duplicated or pre-empted (mirrors create-edge.ts's documented approach).

Acceptance criteria:

  • Valid input produces a successful store call and returns the created signal.
  • Missing/blank required fields → tool-level validation error before any store call.
  • Store 4xx (e.g. unreviewable branch status, unknown branch) is surfaced as-is, not swallowed or rewritten.

Verification: pnpm test:mcp

SG5 — Docker end-to-end verification done

Layer: full stack, containerized   Depends on: sg2-notification-fanout, sg3-notification-consumption, sg4-mcp-submit-verification-signal

Objective: Prove the full submit-signal → fan-out → list/read-notification loop against the real running system.

  • Bring the system up with docker compose up --build spoolstore (per apps/store/AGENTS.md).
  • Obtain a real human session token via pnpm dev:session-token (drives the actual GET /auth/github/login → GET /auth/github/callback flow against the running container, using the github-oauth-stub service in compose.yaml). Do not substitute a hand-crafted token, a client-supplied actor field, or any auth bypass — GET /notifications and POST /notifications/:id/read are human-only and must be exercised with this real token for both the success path and the negative case (missing/invalid token). The cross-stakeholder 404 case (SG3's acceptance criteria) is verified at the store test level via directly-seeded rows, not required in this Docker exercise (the containerized OAuth stub only backs a single fixture stakeholder).
  • Create a branch and submit it (existing G02/G04 flows) so it reaches submitted status.
  • Submit a verification signal via the submit-verification-signal MCP tool specifically (not a bare HTTP call) against the running store — required to prove the MCP integration path.
  • Confirm via a direct DB check or GET /branches/:id/verification-signals that the signal was recorded, and that the branch's status is still submitted (no auto-transition).
  • Using the human session token, call GET /notifications and confirm at least one unread notification referencing that signal/branch exists.
  • Mark it read via POST /notifications/:id/read with the human token; re-fetch GET /notifications?status=unread and confirm it no longer appears.
  • Confirm GET /notifications and POST /notifications/:id/read without a valid human session token are rejected with 401 against the running container.

Acceptance criteria: all bullets above observed against the containerized system (not unit/integration tests alone).

Verification: manual/scripted HTTP + MCP calls against the Dockerized spoolstore, using a token minted by pnpm dev:session-token.
