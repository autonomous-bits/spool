G11 — Workspace Isolation & Request Scoping done

Size: 5/5 — new workspace_id column + backfill + re-keyed unique indexes across eight existing tables, a new session-token claim plus a whole extra OAuth bootstrap path, a two-tier domain-enforcement amendment, and wiring through every existing controller/repository/MCP tool. No new domain concept is invented (Workspace/Membership already exist from G10), but the blast radius across every existing layer is total.

Scope framing: this goal completes the isolation/enforcement work G10 explicitly deferred (see apps/store/src/persistence/migrations/0012_add_workspaces_tables.sql comment: "Meridian IDEA-90 ... is promoted but explicitly out of scope for this goal"). It does not add any new product concept — Workspace/WorkspaceMembership, POST /workspaces, and POST /workspaces/:id/members already exist and are reused unmodified except for the new header check added here.

Sub-goal Detail

SG1 — Persistence: workspace_id + backfill + indexes done

Layer: Persistence  |  Depends on: none

Objective: Migration adding a nullable-then-backfilled-then-NOT-NULL workspace_id UUID REFERENCES workspaces(id) column to chunks, edges, branches, artifacts, chunk_artifacts, suggestions, verification_signals, feedback_notifications, per IDEA-90. Create one implicit default workspace (per IDEA-91) and backfill every pre-existing row in those tables to it before adding the NOT NULL constraint. Re-key existing uniqueness/lookup indexes (e.g. chunk label uniqueness) to be composite on (workspace_id, ...) instead of global.

Acceptance criteria:

  • New migration file (next sequential number after 0012) adds workspace_id to all eight named tables.
  • A single default workspace row is created (or reused if one already exists) and every pre-existing row in the eight tables is backfilled to it in the same migration, before any NOT NULL constraint is applied — matching IDEA-91's "existing stakeholders/branches continue to function unchanged" requirement.
  • Chunk label uniqueness (and any other global-uniqueness index across the eight tables) is dropped and replaced with a composite index/constraint including workspace_id, so two workspaces may each contain a chunk labeled identically without collision.
  • Existing repository queries in apps/store/src/persistence/*.repository.ts are NOT yet changed to filter by workspace_id in this sub-goal (that is SG4/SG5) — SG1 is schema-only, but must not break any existing repository test (i.e. the backfilled default workspace keeps current tests passing unmodified).

Verification: pnpm --filter store build and pnpm test:store (existing repository/migration tests still green against the new schema).

SG2 — Auth: token claim + OAuth bootstrap flow done

Layer: Auth  |  Depends on: none

Objective: Extend SessionTokenClaims with an optional workspaceId: string | null claim (per IDEA-92). Extend GET /auth/github/login to accept an optional workspaceId query param, and GET /auth/github/callback/AuthService.handleCallback to: (a) if workspaceId given, validate via WorkspaceRepository.isMember and mint a workspace-bound token, rejecting with 403/appropriate error if not a member; (b) if omitted and the stakeholder has zero memberships, mint a workspace-less bootstrap token (per IDEA-101); (c) if omitted and the stakeholder has ≥1 membership, reject with 400.

Acceptance criteria:

  • SessionTokenClaims.workspaceId is string | null; a null/absent claim is the workspace-less bootstrap variant.
  • GET /auth/github/login?workspaceId=... carries the param through the OAuth state round-trip to the callback (mirroring how OAuthStateService already round-trips state).
  • Callback mints a workspace-bound token only after confirming membership via WorkspaceRepository.isMember; mints a workspace-less token only when the stakeholder has zero workspace_memberships rows; returns 400 for a member with no workspaceId param, per IDEA-101.
  • tools/dev-session-token/get-dev-session-token.mjs is updated to accept a WORKSPACE_ID env var / CLI flag, so SG7's Docker e2e can obtain both bootstrap and workspace-bound tokens through the real login/callback flow.
  • Unit tests cover: bound-token mint on valid membership, bootstrap-token mint on zero memberships, 403/400 rejection paths.

Verification: pnpm test:store (targeted: auth.service.spec.ts, session-token.service.spec.ts, auth.controller.spec.ts)

SG3 — Domain: two-tier workspace-scope assertion done

Layer: Domain  |  Depends on: SG1, SG2

Objective: Add a reusable assertion (e.g. assertWorkspaceScope, alongside the existing branch-lifecycle.ts assertions per IDEA-70's precedent that enforcement lives in domain/persistence, not a gateway) implementing IDEA-98's amendment via IDEA-100's two-tier rule: for session-token-gated routes, assert the request's X-Workspace-Id header equals the verified token's workspaceId claim; for delegated/tokenless routes, assert X-Workspace-Id matches a workspace_memberships row for the caller-supplied stakeholderId (via SG1's schema + existing WorkspaceRepository.isMember).

Acceptance criteria:

  • One exported assertion function taking either a verified token's workspaceId claim or a caller-supplied stakeholderId, plus the request's X-Workspace-Id header value, and throwing a distinct domain error (e.g. WorkspaceScopeViolationError) on mismatch or missing membership.
  • Missing X-Workspace-Id header is itself a rejection (per IDEA-92 "every ... request must include" the header) — not treated as "no scoping requested".
  • Unit tests cover: token-claim match/mismatch, delegated-membership match/no-membership, missing header.

Verification: pnpm test:store (targeted: new domain spec file, e.g. workspace-scope.spec.ts)

SG4 — API core-graph: chunks/edges/branches/workspaces done

Layer: API  |  Depends on: SG3

Objective: Wire SG3's assertion into ChunksController/service/repository, EdgesController/service/repository, BranchesController/service/repository (create/submit/verify/reject/merge/get), and WorkspacesController's POST /workspaces/:id/members (which now also requires X-Workspace-Id to equal :id, per IDEA-92's blanket "every request" rule). POST /workspaces itself is the sole exception — it targets no existing workspace and accepts a bootstrap token with no workspaceId claim, so it does not require the header. All repository queries for these four modules add a workspace_id filter/column.

Acceptance criteria:

  • Chunk/edge/branch capture, read, and lifecycle-transition endpoints all require and validate X-Workspace-Id via SG3's assertion; requests missing it, mismatched, or from a non-member are rejected (401/403 per existing error-mapping conventions).
  • Chunk labels are unique per workspace end-to-end: creating "IDEA-1"-labelled chunks (or this repo's chunk-label equivalent) in two different workspaces both succeed.
  • POST /workspaces/:id/members requires X-Workspace-Id to equal the path's :id; POST /workspaces requires no header and accepts a bootstrap token.
  • All chunk/edge/branch repository reads exclude rows from other workspaces (no cross-workspace label resolution, per IDEA-90).

Verification: pnpm test:store (targeted: chunks/edges/branches/workspaces controller + repository specs)

SG5 — API secondary + artifact blob-key prefixing done

Layer: API/Persistence  |  Depends on: SG4

Objective: Apply the same SG3 assertion + workspace_id filtering to ArtifactsController (create, chunk-artifact attach/list, download-token issuance), SuggestionsController, VerificationSignalsController, and NotificationsController. Additionally implement IDEA-93: prefix every ArtifactBlobStore key with {workspaceId}/ (e.g. {workspaceId}/{artifactId}/{filename}) in the existing local-file blob store, no new store instance.

Acceptance criteria:

  • POST /artifacts, POST /chunks/:label/artifacts, GET /chunks/:label/artifacts, and GET /artifacts/:id/download-token all require and validate X-Workspace-Id per SG3 (delegated-tokenless tier, per IDEA-100), and reference chunks by label scoped to that same workspace (reusing SG4's workspace-scoped chunk lookup).
  • GET /artifacts/content/:token is explicitly exempt from the X-Workspace-Id header check: the signed download token is already cryptographically bound to one specific artifact (and thus one workspace) at issuance time, so no additional caller-supplied header is meaningful at redemption time. Record this as a deliberate, documented exemption, not a silent gap.
  • Artifact blob storage keys are prefixed by workspaceId per IDEA-93; existing (pre-migration) artifacts are re-keyed or the store resolves their key via the default workspace from SG1's backfill (document whichever approach is chosen).
  • SuggestionsController (create/accept/reject/list/get), VerificationSignalsController (create/list), and NotificationsController (list/mark-read) all require and validate X-Workspace-Id — token-gated tier for suggestions accept/reject and notifications (existing human-session-token routes), delegated tier for suggestion create and verification-signal create.

Verification: pnpm test:store (targeted: artifacts/suggestions/verification-signals/notifications controller + repository specs, plus blob-store key-prefix spec)

SG6 — MCP: pass workspaceId through all 7 tools done

Layer: MCP  |  Depends on: SG5

Objective: Extend all 7 existing MCP tools (capture-chunk, create-branch, create-edge, submit-suggestion, submit-verification-signal, attach-artifact-to-chunk, upload-artifact) to accept a workspaceId input and set the X-Workspace-Id header on the corresponding store HTTP call, per IDEA-92's "every ... MCP tool call must include" requirement.

Acceptance criteria:

  • Each of the 7 tools' input schema gains a required workspaceId field.
  • Each tool sets X-Workspace-Id: on its outbound store request.
  • A store rejection (missing/mismatched workspace scope) surfaces as a clear MCP tool error, not a generic failure.

Verification: pnpm test:mcp

SG7 — Docker end-to-end (final, mandatory) done

Layer: Docker e2e  |  Depends on: SG6

Objective: Bring the system up with docker compose up --build spoolstore (per apps/store/AGENTS.md) and exercise the full workspace-isolation flow over real HTTP against the running containerized store, using session tokens obtained ONLY via pnpm dev:session-token (SG2's updated script) driving the real GET /auth/github/login → GET /auth/github/callback flow against the github-oauth-stub service already wired into compose.yaml. Never substitute a hand-crafted or manually-signed token, or a client-supplied workspaceId/claim field, for any scenario below — the real login/callback flow via the tool is the only valid way to obtain each token.

Acceptance criteria (all against the running container):

  • A stakeholder with zero workspace memberships obtains a workspace-less bootstrap token via pnpm dev:session-token; it succeeds against POST /workspaces and is rejected (403) against every other session-token-gated route.
  • After creating a workspace, re-running pnpm dev:session-token with that workspaceId yields a workspace-bound token.
  • A chunk capture request (delegated/tokenless route) with X-Workspace-Id matching the caller-supplied stakeholder's membership succeeds; the same request with a non-member/mismatched X-Workspace-Id is rejected.
  • Two chunks with the same label, created in two different workspaces, both succeed (per IDEA-90's per-workspace label uniqueness) and each workspace's read only sees its own chunk.
  • A token-gated route (e.g. branch submit) with a workspace-bound token whose X-Workspace-Id header does not match the token's workspaceId claim is rejected.
  • An artifact uploaded in one workspace is stored under a {workspaceId}/...-prefixed key and is retrievable via its own download-token flow.

Verification: Manual/scripted HTTP exercise against docker compose up --build spoolstore, using pnpm dev:session-token for every token — not unit/in-process tests alone.
