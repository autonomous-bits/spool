Sub-goal detail

SG0 — API + persistence: GitHub OAuth login/callback + session-token minting done

Depends on: none

Objective: Implement the concrete human-authentication mechanism ratified by IDEA-81: a completed GitHub OAuth 2.0 authorization-code grant mints a short-lived, store-issued session token.

  • GET /auth/github/login redirects to github.com/login/oauth/authorize with client_id/redirect_uri/a generated anti-CSRF state value.
  • GET /auth/github/callback?code&state validates state, exchanges code for a GitHub access token via POST https://github.com/login/oauth/access_token, then resolves the GitHub identity via GET https://api.github.com/user.
  • Both GitHub-facing HTTP calls go through an injectable HTTP client (e.g. a GithubOAuthClient port), so SG5's Docker e2e can substitute a local stub implementing the same two endpoints instead of driving a live, interactive github.com consent screen.
  • New stakeholders.github_login column (migration) plus StakeholderRepository.findByGithubLogin(login): Promise<{ id: string; discipline: Discipline | null } | undefined>. 400/401 if the resolved GitHub login has no matching stakeholder.
  • SessionTokenService.sign({ stakeholderId, discipline, authTime }) mints a short-lived (e.g. 15 min) signed token (JWT via @nestjs/jwt or jose, server-held secret from config, never the client); SessionTokenService.verify(token) returns the claims or throws on expiry/invalid signature.
  • Callback response returns the session token to the caller (JSON body or Set-Cookie); exact transport is repo-derived but must be usable as a Bearer token by SG4.
  • Unit/integration tests cover: successful callback mints a token whose claims round-trip through verify; unknown GitHub login is rejected; invalid/missing state is rejected; expired token fails verify.

Verification: pnpm test:store (new auth module unit + integration tests).

SG1 — domain: ActorKind/ActorContext + branch-lifecycle assertions done

Depends on: none

Objective: Model the human/delegated actor distinction and the submission invariants as pure, testable domain functions, per IDEA-75/IDEA-35/IDEA-53.

  • ActorKind closed vocabulary: 'human' | 'delegated', with isActorKind/parseActorKind guards mirroring existing vocab modules (e.g. branch-status.ts).
  • ActorContext type: { kind: ActorKind; stakeholderId: string; discipline: Discipline }. HumanActorContext/DelegatedActorContext are the kind-narrowed variants (discriminated union), per IDEA-75's exact naming.
  • assertIsHumanActor(actor: ActorContext): asserts actor is HumanActorContext — throws a typed domain error (e.g. ActorAuthorizationError) when actor.kind !== 'human'.
  • assertSubmitDiscipline(actor: ActorContext, branch: Branch): void — throws a typed domain error when actor.discipline !== branch.discipline.
  • assertDraftStatus(branch: Branch): void — throws a typed domain error when branch.status !== 'draft'.
  • These three assertions are pure functions with no persistence access; the caller (SG4) is responsible for constructing ActorContext from SG0's verified session-token claims (stakeholderId/discipline), never a client-supplied discipline or a body-declared actorKind.
  • Unit tests cover: each assertion's pass/throw cases, and that ActorKind rejects any value outside the closed set.

Verification: pnpm test:store (new domain unit tests).

SG2 — persistence/infra: StakeholderRepository + atomic branch transition done

Depends on: none

Objective: Load a stakeholder's discipline server-side (never trust a client-supplied discipline for authorization) and persist the draft→submitted transition without a read-then-write race.

  • New StakeholderRepository.findById(id): Promise<{ id: string; discipline: Discipline | null } | undefined> (mirrors ChunkRepository/BranchRepository query style; stakeholders.discipline is nullable per the existing schema, so the type reflects that).
  • BranchRepository.submit(branchId: string): Promise<Branch | undefined> — a single atomic statement: UPDATE branches SET status = 'submitted', submitted_at = now(), updated_at = now() WHERE id = $1 AND status = 'draft' RETURNING *. Returns undefined when zero rows match (covers both "not found" and "not currently draft" — the caller disambiguates via a preceding findById used only for the 404/409 error message, not as the authorization check itself).
  • This atomic UPDATE ... WHERE status = 'draft' pattern is the reference pattern SG3 retrofits onto chunk/edge writes.
  • Integration tests (containerized Postgres, per repo convention) cover: StakeholderRepository.findById round-trip including a stakeholder with discipline IS NULL; BranchRepository.submit succeeds exactly once for a draft branch and returns undefined on a second concurrent/sequential call (status no longer 'draft'); submitted_at is persisted and round-trips.

Verification: pnpm test:store (repository e2e tests against containerized Postgres).

SG3 — persistence: harden chunk/edge writes against submitted branches done

Depends on: SG2

Objective: Close the race where a chunk/edge write lands between ChunksService/EdgesService's pre-read branch-status check and the insert, so submission actually locks the branch (current check-then-insert in chunks.service.ts/edges.service.ts is not atomic).

  • ChunkRepository.create and EdgeRepository.create are changed to guard the insert against the branch's current status inside the same statement/transaction, e.g. wrapping in a client transaction that does SELECT status FROM branches WHERE id = $1 FOR UPDATE (only when a branchId is present) before the insert, rolling back and signalling a conflict if status is no longer 'draft'.
  • ChunksService/EdgesService's existing pre-read status check remains (fast-path, unchanged error shape for the common case) but is no longer the sole enforcement — the repository-level guard is what actually prevents the race.
  • A conflict raised by the repository-level guard surfaces as the same ConflictException/409 shape the services already throw for a non-draft branch, so no API-visible behavior changes for the non-racing case.
  • Integration tests (containerized Postgres) demonstrate the race is closed: a branch is submitted concurrently with a chunk (and separately an edge) creation targeting the same branch, and exactly one of the two operations succeeds — the write path branch-status is never split by a stale in-process read.

Verification: pnpm test:store (repository e2e tests, including the concurrency case).

SG4 — API: POST /branches/:id/submit done

Depends on: SG0, SG1, SG2

Objective: Expose the submission transition over HTTP, requiring a valid GitHub-OAuth-derived session token (SG0) rather than a caller-declared actor field, wiring SG1's assertions and SG2's persistence together.

  • POST /branches/:id/submit requires Authorization: Bearer <session-token>; no actorKind/discipline is accepted in the request body.
  • 401 if the header is missing, malformed, or SessionTokenService.verify rejects it (invalid signature, expired, or authTime outside a bounded max-age).
  • 404 if the branch does not exist.
  • 400 if the token's stakeholderId does not resolve via StakeholderRepository.findById, or resolves with discipline IS NULL (a stakeholder with no assigned discipline cannot satisfy the discipline-match invariant).
  • The service constructs a HumanActorContext directly from the verified token's claims (a valid session token can only ever represent a human actor per IDEA-81, so assertIsHumanActor becomes a structural guarantee of SG0's token issuance rather than a runtime branch on client input) — delegated/agent callers have no path to a session token and so 401 before reaching this check.
  • Calls assertSubmitDiscipline(actor, branch): 409 if the stakeholder's discipline does not match the branch's discipline.
  • Calls assertDraftStatus(branch) as a fast-path check, then BranchRepository.submit(branchId) as the atomic guard: 409 if the branch is not (or no longer) in 'draft' status.
  • BranchResponse gains a nullable submittedAt field, populated on GET /branches/:id and the submit response.
  • e2e tests cover: successful submission by a matching-discipline human actor with a valid session token; 401 (missing/invalid/expired token); 400 (unknown stakeholder / null-discipline stakeholder); 404 (unknown branch); 409 (discipline mismatch); 409 (branch already submitted).

Verification: pnpm test:store (controller/service unit tests + e2e spec against containerized Postgres).

SG5 — Docker end-to-end verification done

Depends on: SG0, SG3, SG4

Objective: Prove the full slice (GitHub OAuth → API → domain → persistence, including the hardened write-lock) works against the real containerized system, not just in-process tests.

  • docker compose up --build spoolstore brings up the store against real Postgres, with SG0's GithubOAuthClient pointed at a local stub standing in for github.com's token-exchange/user endpoints (a live interactive GitHub consent screen cannot be automated in this e2e run; this substitution is scoped explicitly, not silently faked).
  • Over real HTTP: drive GET /auth/github/login → stubbed callback → GET /auth/github/callback for a GitHub login mapped to the repo's known-valid bootstrap stakeholder fixture (non-null discipline matching the branch), obtaining a real minted session token.
  • Create a branch (POST /branches) using that same stakeholderId; submit it (POST /branches/:id/submit) with Authorization: Bearer <session-token> succeeds, and GET /branches/:id reflects status: 'submitted' and a non-null submittedAt.
  • Negative case over HTTP: submitting the same branch again returns 409.
  • Negative case over HTTP: submitting with no Authorization header (i.e. no completed GitHub OAuth flow) returns 401.
  • Negative case over HTTP: after submission, POST /chunks (and separately POST /edges) targeting that branch is rejected (409), confirming the write-lock retrofit from SG3 holds against the real running system, not only in-process tests.

Verification: manual/scripted HTTP exercise (curl or equivalent) against spoolstore, covering the success path and all four negative cases above — not satisfied by unit/in-process tests alone.
