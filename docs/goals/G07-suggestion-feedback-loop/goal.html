Sub-goal Detail

SG1 — submit-suggestion done

Layer: domain + persistence + API + MCP (full vertical slice)   Depends on: none

Objective: External/delegated actors can submit a chunk- or edge-modification suggestion that lands as a pending row, over both HTTP and the MCP tool.

Design decisions (field-level shape is ratified by IDEA-31 → schema.sql's suggestions/suggestion_state_logs tables, ADR promoted — not free-form; only the additive submitter-attribution columns below are a documented scope deviation, mirroring the IDEA-82 precedent):

  • SuggestionStatus vocabulary: 'pending' | 'accepted' | 'rejected' (IDEA-27, matches schema.sql's status CHECK).
  • Suggestion row exactly matches schema.sql's suggestions table: id, label VARCHAR(50) (chunk-suggestion only), content TEXT (chunk-suggestion only), from_chunk_label/to_chunk_label VARCHAR(50) (edge-suggestion only), relationship_type (edge-suggestion only, same closed vocab as edges.type), discipline (closed Discipline vocab, NOT NULL — the discipline the resulting feedback branch will be scoped to, per IDEA-28), status, created_at, updated_at. No kind/summary/payload columns — the discriminated-union CHECK constraint check_suggestion_type (either label+content set with from/to/relationship_type NULL, or the reverse) is the sole discriminator; enforce it both at the DB (migration) and API validation layers. Unique index idx_suggestions_unique (label, content) per schema.sql.
  • Documented scope deviation (mirrors IDEA-82's precedent of extending schema.sql for real implementation needs; escalated upstream as promoted Meridian feedback IDEA-83, feedback-on IDEA-31/IDEA-49/IDEA-75/IDEA-82 — schema.sql itself not yet physically edited, this deviation proceeds regardless): add submitted_by_stakeholder_id (NOT NULL, FK stakeholders) and submitted_by_actor_kind (server-assigned, never client-supplied — IDEA-75) columns, since schema.sql's suggestions table has no submitter-attribution column (unlike branches.created_by_stakeholder_id NOT NULL) and IDEA-75 requires distinguishing delegated vs. human actors. decided_by_stakeholder_id/decided_at are also additive (nullable) for the same reason SG2/SG3 need to know who accepted/rejected — schema.sql tracks this only via suggestion_state_logs.updated_by_stakeholder_id, which this goal also uses, so these two columns are optional convenience denormalization; keep them nullable and non-blocking if dropped in review.
  • Table suggestion_state_logs matches schema.sql exactly: id, suggestion_id FK, old_status (nullable for the initial insert), new_status, updated_by_stakeholder_id FK stakeholders, created_at — satisfies IDEA-27's "all suggestion state changes are logged". No actor_kind column (not in schema.sql); actor-kind enforcement (IDEA-75) happens at the API layer via assertIsHumanActor, not persisted per-event. Every status change (submit/accept/reject) inserts exactly one row here in the same transaction as the status update.
  • Migration 0008_add_suggestions_table.sql: creates suggestions and suggestion_state_logs per schema.sql plus the two additive submitter columns above; does not re-add branches.origin_suggestion_id (it already exists as a plain nullable UUID per migration 0003's documented scope deviation) — instead adds the FK constraint branches_origin_suggestion_id_fkey referencing suggestions(id).
  • POST /suggestions: delegated-actor style (mirrors capture-chunk/create-branch/create-edge) — body carries stakeholderId plus exactly one variant's fields (chunk: label+content; edge: fromChunkLabel+toChunkLabel+relationshipType) and discipline; no session token required; server sets submitted_by_actor_kind = 'delegated' unconditionally (never trusts a client-supplied actor kind, per IDEA-75).
  • MCP tool submit-suggestion (new file apps/mcp/src/tools/submit-suggestion.ts): validates input, proxies to POST /suggestions, structured exactly like capture-chunk.ts (own parse*Input, *ValidationError, result type, spec file).

Acceptance criteria:

  • POST /suggestions with a valid chunk-shaped body (label, content, discipline, stakeholderId) OR a valid edge-shaped body (fromChunkLabel, toChunkLabel, relationshipType, discipline, stakeholderId) persists a suggestion with status='pending' and inserts one suggestion_state_logs row (old_status=NULL, new_status='pending').
  • A body mixing both variants' fields, or missing all of both variants' fields, → 400 (mirrors check_suggestion_type); unknown stakeholderId → 400 (mirrors existing chunk/branch/edge creation error handling).
  • SG1 does not require a read endpoint; assert the persisted row directly via the repository/DB in tests. GET /suggestions/:id is introduced in SG3, not SG1.
  • MCP submit-suggestion tool: unit tests cover valid chunk-shaped input, valid edge-shaped input, validation failures (mixed/missing variant fields), and store-error propagation (mirrors capture-chunk.spec.ts structure).

Verification: pnpm test:store, pnpm test:mcp

SG2 — accept-suggestion done

Layer: domain + persistence + API (no MCP tool — human-only per IDEA-75 and existing branch-lifecycle precedent)   Depends on: sg1-submit-suggestion

Objective: A human stakeholder (session token, per G04/G05's SessionTokenService) accepts a pending suggestion, atomically creating a linked draft branch.

  • POST /suggestions/:id/accept body: { name: string } (branch name — reuses the same non-empty-string validation as CreateBranchRequest; branch discipline comes from the suggestion row, not the request body).
  • Auth: Authorization: Bearer <session token>, verified via existing SessionTokenService; resolves a HumanActorContext exactly like BranchesService.resolveActorForVerification.
  • Must be one atomic DB transaction: lock the suggestion row (SELECT ... FOR UPDATE), assert status='pending' (else 409, mirrors BranchLifecycleError/ConflictException pattern), insert the new branch row (status=draft, discipline=suggestion.discipline, origin_suggestion_id=suggestion.id, created_by_stakeholder_id=acting human), update the suggestion (status='accepted', decided_by_stakeholder_id, decided_at), and insert one suggestion_state_logs row (pending→accepted) — all-or-nothing.
  • Add originSuggestionId: string | null to BranchResponse/toBranchResponse so the linkage is publicly visible (currently missing from the DTO).
  • Response: the created BranchResponse (with originSuggestionId set) — not the suggestion.

Acceptance criteria:

  • Accepting a pending suggestion creates exactly one new branch with status='draft', correct discipline, and originSuggestionId equal to the suggestion's id.
  • Accepting a non-pending suggestion (already accepted/rejected) → 409, no branch created, no transition row inserted.
  • Accepting with a non-human/invalid/missing session token → 401 (mirrors existing branch-lifecycle auth tests).
  • A failure injected mid-transaction (e.g. duplicate branch name) rolls back both the suggestion status and any partial branch row — no orphaned state.
  • Exactly one suggestion_state_logs row is inserted per successful accept.

Verification: pnpm test:store

SG3 — reject + list/read suggestions done

Layer: domain + persistence + API (no MCP tool)   Depends on: sg2-accept-suggestion

Objective: Human stakeholders can reject a pending suggestion, and any caller can list/read suggestions for review (closing the "queue" aspect of IDEA-28/IDEA-49).

  • POST /suggestions/:id/reject: same auth/actor pattern as accept; asserts status='pending' (409 otherwise); sets status='rejected', decided_by_stakeholder_id, decided_at; inserts one suggestion_state_logs row (pending→rejected); no branch created.
  • GET /suggestions/:id: returns the suggestion (404 if missing).
  • GET /suggestions?status=pending (optional query param, defaults to no filter): returns the list, ordered by created_at ascending — this is the human-review "queue" view.
  • Explicit decision: GET /suggestions and GET /suggestions/:id require no auth (unauthenticated read), matching the existing precedent of GET /branches/:id/GET /chunks/:id/GET /edges/:id, which are also unauthenticated in this codebase. This is a deliberate consistency choice, not an oversight.

Acceptance criteria:

  • Rejecting a pending suggestion sets status='rejected' and logs one transition row; rejecting a non-pending suggestion → 409.
  • GET /suggestions?status=pending returns only pending suggestions, oldest first.
  • GET /suggestions/:id returns 404 for an unknown id.
  • GET /suggestions and GET /suggestions/:id succeed with no Authorization header (test explicitly omits it) — confirming the deliberate unauthenticated-read decision above is actually implemented, not just documented.

Verification: pnpm test:store

SG4 — Docker end-to-end verification done

Layer: full stack, containerized   Depends on: sg3-reject-list-suggestions

Objective: Prove the full submit → accept/reject → linked-branch loop against the real running system, not in-process tests.

  • Bring the system up with docker compose up --build spoolstore (per apps/store/AGENTS.md).
  • Obtain a real human session token via pnpm dev:session-token (drives the actual GET /auth/github/login → GET /auth/github/callback flow against the running container, using the github-oauth-stub service in compose.yaml). Do not substitute a hand-crafted token, a client-supplied actor field, or any auth bypass — accept/reject are human-only and must be exercised with this real token for both the success path and negative cases (wrong/missing token, already-decided suggestion).
  • Submit the first suggestion via the submit-suggestion MCP tool specifically (not a bare HTTP call) against the running store — this is required to prove the MCP integration path, not optional.
  • Accept it using the human session token; confirm via GET /branches/:id that a new branch exists with the correct discipline and originSuggestionId.
  • Submit a second suggestion (either via the MCP tool or a direct POST /suggestions call) and exercise the reject path with the human token; confirm its status is rejected via GET /suggestions/:id.
  • Confirm accept/reject without a valid human session token is rejected with 401 against the running container.

Acceptance criteria: all six bullets above observed against the containerized system (not unit/integration tests alone).

Verification: manual/scripted HTTP + MCP calls against the Dockerized spoolstore, using a token minted by pnpm dev:session-token.
